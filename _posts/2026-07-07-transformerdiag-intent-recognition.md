> 一句话总结：TransformerDiag 的意图识别不是简单把用户问题分成几个标签，而是在电力变压器诊断场景里，用“硬模式判定 + 会话改写 + DST/Context 状态管理 + UniQA 工具编排 + SSE 兜底”构成一条可上线的智能体入口链路。

![TransformerDiag 意图识别主链路](/assets/images/transformerdiag-intent/intent-chain-overview.svg)

<!--more-->

我最近看了一个电力智能体项目：`TransformerDiag`。它的定位不是普通问答机器人，而是面向变压器故障诊断、试验数据分析、缺陷隐患判断和运维知识问答的专业系统。

这个项目里最值得单独拆开的模块，是**意图识别**。

很多人一听“意图识别”，会想到一个很轻的分类器：用户问什么，就分到“问答”“诊断”“推荐”几个类别。但在电力场景里，这样做远远不够。因为一个错误的意图路由，可能会导致系统调用错误工具、遗漏必要数据、把泛化知识回答当成工程处置建议，甚至在没有足够证据时给出过度判断。

所以 TransformerDiag 的意图识别做得更像一个**专业智能体的网关层**：它既要理解用户语义，也要判断是否具备进入某条专业链路的参数条件；既要利用历史上下文，又不能让历史上下文污染当前问题；既要能调用外部工具，也要在工具失败时保持可解释、可降级。

这篇文章就专门拆一下它的设计。

## 一、项目背景：为什么电力智能体不能只做“聊天分类”

TransformerDiag 是一个变压器智能诊断系统，后端以 FastAPI 为核心，通过 SSE 向前端流式推送处理过程。系统内部融合了多类能力：

| 能力模块 | 作用 | 典型数据源或组件 |
| --- | --- | --- |
| 变压器知识问答 | 回答原理、标准、方法、规范类问题 | LLM、RAG 知识工具 |
| 试验数据分析 | 查询和解释历史试验、油色谱、绝缘、电阻等数据 | PostgreSQL、南瑞试验接口 |
| 缺陷隐患判断 | 面向设备状态、缺陷、隐患、风险进行分析 | 专业 prompt、缺陷/隐患工具 |
| 跳闸诊断 | 围绕保护动作、跳闸事件做诊断 | 事件库、小模型、知识图谱 |
| 会话记忆 | 支撑多轮追问、设备继承、上下文改写 | DST、Context、PG 历史、ctxopt |

在这样的系统里，用户的一句话可能对应完全不同的执行路径。

例如：

| 用户问题 | 直觉看起来 | 系统真正需要判断 |
| --- | --- | --- |
| “什么是三比值法？” | 普通知识问答 | 走统一问答，不需要设备、不需要查历史数据 |
| “楠竹塘#3主变油色谱 H2 为 150，正常吗？” | 试验分析 | 满足试验结果解读硬模式，需要抽取试验名和数值 |
| “#1主变套管渗油属于什么缺陷，怎么处理？” | 缺陷分析 | 进入缺陷隐患硬模式，需要工程化处置建议 |
| “最近一次油色谱结果怎么样？” | 试验问答 | 可能需要继承会话里的 transformer_id，并查历史数据 |
| “帮我推荐几个典型跳闸案例” | 案例请求 | 不应误判成具体跳闸诊断 |

这就是 TransformerDiag 意图识别的核心难点：**不是判断一句话“像不像某类问题”，而是判断它是否具备进入某个专业流程的充分条件。**

## 二、总体链路：从 SSE 请求到最终回答

当前项目中，意图识别入口是：

```text
POST /api/v1/diagnosis/intent-recognition/stream
```

它对应的核心编排在 `StandardDiagnosisAdapter.handle_intent_recognition_request_v8()`，底层识别逻辑主要在 `IntentRecognitionService.recognize_intent_v8()`。

整体链路可以拆成 8 步：

| 步骤 | 关键动作 | 工程目的 |
| --- | --- | --- |
| 1 | 接收 SSE 请求，生成或携带 `session_id`、`context_id` | 把本轮问题纳入一个可追踪会话 |
| 2 | 归一化用户输入，保留原始问题 | 区分 `raw_query` 与后续改写问题 |
| 3 | 通过台账或上下文获取 `transformer_id`、`transformer_info` | 让问题绑定到具体设备 |
| 4 | 调用 `ctxopt /api/query` 做问题扩充改写 | 解决多轮追问里的指代、省略问题 |
| 5 | 从 PostgreSQL 取最近三轮问答，只作为意图分类辅助上下文 | 让分类理解历史，但不污染当前问题 |
| 6 | 调用 LLM 进行硬模式意图识别 | 判断试验解读、缺陷隐患、统一问答、未知意图 |
| 7 | 更新 DST/Context/槽位，并路由到 UniQA | 把识别结果转成可执行状态 |
| 8 | UniQA 工具编排、LLM 流式输出、写回 PG 和 ctxopt | 形成可追踪、可连续问答的闭环 |

这条链路里有一个很重要的设计点：**当前问题和历史上下文是分开的。**

项目文档中明确说明，`ctxopt` 请求体里的 `query` 只使用当前最新问题，不拼接 PostgreSQL 历史上下文；PG 最近三轮问答只用于意图识别辅助。这样做可以避免一个常见问题：用户明明问的是“什么是三比值法”，结果因为上一轮聊过某台主变，系统把当前知识问答误改写成某台设备的试验分析。

## 三、意图类型：不是越多越好，而是边界要硬

代码里保留了比较多的历史意图类型，例如跳闸诊断、知识问答、主变历史试验问答、案例推荐、试验结果解读、隐患、缺陷、统一问答、未知意图等。

但 V8 链路的实际硬判定重点，已经收敛到下面几类：

| 意图 | 类型 | 说明 | 后续链路 |
| --- | --- | --- | --- |
| 试验结果解读 | 硬模式 | 用户已经给出具体试验项目和数值，要求判断、分析、评价 | UniQA + 试验解读逻辑 |
| 缺陷隐患 | 硬模式 | 用户围绕具体设备状态、缺陷/隐患/故障关键词和处置决策提问 | UniQA + 缺陷/隐患分析 |
| 统一问答 | 非硬模式 | 电力变压器相关原理、规范、方法、运维经验、案例性说明 | UniQA + 知识工具 |
| 未知意图 | 兜底边界 | 非电力、闲聊、身份、商务采购等 | 固定引导回答 |

这个收敛很关键。很多系统会把意图类型做得很细，最后导致边界互相打架。TransformerDiag 的思路更实用：**只有确实需要进入工程链路、且参数满足条件的，才进入硬模式；否则尽量归到统一问答。**

## 四、硬模式：用“四要素”防止模型过度推断

![TransformerDiag 硬模式意图判定矩阵](/assets/images/transformerdiag-intent/hard-mode-matrix.svg)

TransformerDiag 的 `intent_prompt.md` 里有一个非常明显的设计思想：让 LLM 只做“硬模式入口判断”，不负责回答、不负责补全、不负责业务决策。

### 1. 试验结果解读的四要素

要判定为“试验结果解读”，必须同时满足：

| 要素 | 要求 | 例子 |
| --- | --- | --- |
| 变电站名称 | 具体站名，不能是“某主变” | 楠竹塘、长沙站 |
| 主变编号 | 明确编号 | #1、#3、1号主变 |
| 试验名称 | 具体试验项目，不能是泛称 | 油色谱、介损、绕组直流电阻 |
| 试验结果数值 | 用户原文直接给出数字 | H2=150、tanδ=0.32%、80MΩ |

如果用户只是问“最近一次油色谱结果怎么样”，虽然它是试验相关问题，但没有直接给出数值，所以不应该进入“试验结果解读”硬模式，而应该走需要查历史数据或统一问答的路径。

这个规则能防止一种危险误判：模型看到“油色谱”“异常”“分析”就直接给出处置建议，但实际上用户并没有提供数据，也没有完成查询。

### 2. 缺陷隐患的成立条件

缺陷隐患也不是看到“严重吗”“有影响吗”就进入。它需要同时具备：

| 条件 | 说明 |
| --- | --- |
| 具体设备 | 主变、某台变压器、具体部件 |
| 可感知状态或现象 | 漏油、发热、放电、油位下降、卡涩等 |
| 明确关键词 | 缺陷、隐患、故障、风险 |
| 工程决策目标 | 判断依据、缺陷性质、处置措施、怎么办 |

这背后是一个很工程化的判断：**缺陷/隐患分析不是泛化常识问答，而是可能影响检修策略和风险研判的专业链路。**

如果用户只是问“套管故障有哪些类型”，这更像知识问答；如果用户说“#1 主变套管渗油，这算什么缺陷，怎么处置”，才更像缺陷硬模式。

### 3. 统一问答承担“非硬模式”的大部分问题

统一问答不是低级兜底，而是系统的主干问答能力。它覆盖：

| 问题类型 | 示例 |
| --- | --- |
| 原理机制 | “三比值法是什么？” |
| 标准规范 | “油色谱乙炔超标一般怎么看？” |
| 运维经验 | “有载调压开关滑档可能有什么影响？” |
| 方法流程 | “如何判断铁芯多点接地？” |
| 案例性说明 | “有哪些典型套管隐患案例？” |

这样做的好处是：只要硬模式条件不完整，但问题仍属于电力变压器技术范围，系统不会拒答，也不会强行走专业工具链，而是用统一问答给出较稳妥的知识性回答。

## 五、`ctxopt`：解决多轮追问里的省略和指代

电力诊断对话里，用户经常不会每轮都把设备和上下文说完整。

第一轮可能是：

```text
楠竹塘#3主变油色谱 H2 为 150，正常吗？
```

第二轮可能只问：

```text
那乙炔呢？
```

如果系统只看第二句话，很难判断“那”指什么，“乙炔”属于哪个试验、哪台设备、哪个上下文。

因此 TransformerDiag 在意图识别前调用 `ctxopt /api/query` 做问题扩充改写。成功时，`rewritten_query` 会替换为扩展后的当前问题；失败时继续使用当前最新问题。

这里有一个很细的工程控制：**ctxopt 只改写当前问题，不把 PG 历史直接拼到当前问题字段里。**

也就是说：

| 字段 | 作用 | 是否进入当前问题 |
| --- | --- | --- |
| `raw_query` | 用户原始输入 | 保存，不覆盖 |
| `cleaned_input` | 空白归一化后的输入 | 作为基础问题 |
| `rewritten_query` | ctxopt 扩充后的问题 | 用作当前问题 |
| PG 最近三轮历史 | 辅助意图分类 | 不写回当前问题 |

这能防止 prompt 里上下文重复出现，也能避免知识工具拿到一段混杂历史的超长问题。

## 六、DST 与 Context：让意图识别跨轮次稳定工作

![TransformerDiag 状态、槽位与记忆闭环](/assets/images/transformerdiag-intent/state-memory-loop.svg)

TransformerDiag 的意图识别不是一次性函数调用，而是围绕会话状态运行。

其中两个对象很关键：

| 对象 | 主要字段 | 作用 |
| --- | --- | --- |
| DST | `session_id`、`transformer_id`、`transformer_info`、`current_intent`、`filled_slots` | 管理会话级状态和设备绑定 |
| Context | `context_id`、`original_input`、`user_input`、`assistant`、`current_intent`、`filled_slots` | 管理单轮对话上下文 |

如果本轮没有显式传入 `transformer_id`，系统会尝试：

1. 从用户输入里抽取设备名称并映射台账。
2. 如果抽取不到，从 DST 继承历史 `transformer_id`。
3. 如果识别到新设备，则更新 DST，并切换到新的设备上下文。

这就是为什么它适合电力运维对话。真实用户不会每一句都重复“楠竹塘#3主变”，但系统必须知道当前问题到底延续哪台设备。

同时，Context 中保存了两个问题版本：

| 字段 | 内容 |
| --- | --- |
| `original_input` | 用户原始问题 |
| `user_input` | 当前改写问题，即 UniQA 实际使用的问题 |

这个设计很实用：日志、审计和用户回放可以看到原始输入；模型调用和工具执行则使用更完整的改写输入。

## 七、LLM 负责语义判定，但不被允许“脑补参数”

项目里一个很好的实践，是把 LLM 的职责压得很窄。

`intent_prompt.md` 反复强调：

| LLM 可以做 | LLM 不应该做 |
| --- | --- |
| 判断是否满足硬模式 | 不回答用户问题 |
| 输出意图 JSON | 不补全缺失设备、试验名、数值 |
| 根据规则裁决冲突 | 不根据常识猜测用户真实意图 |
| 将要素不足的问题降级到统一问答 | 不把“像缺陷”推断成缺陷 |

这对垂直行业智能体非常重要。

LLM 很擅长语义理解，但也容易“帮用户补完没说的话”。在娱乐或普通问答里，这可能只是表达更自然；在电力诊断里，这会变成风险。TransformerDiag 通过硬模式规则把 LLM 约束在入口判定上，能显著降低过度推断。

## 八、意图识别后的路由：多类问题统一进入 UniQA

当识别结果属于以下类型时，系统会进入 `_handle_uniqa_flow_v8()`：

| 意图 | 是否进入 UniQA | 原因 |
| --- | --- | --- |
| 统一问答 | 是 | 需要知识问答、RAG 或通用 LLM |
| 试验结果解读 | 是 | 需要结合试验参数构造专业分析 |
| 隐患 | 是 | 需要隐患风险分析链路 |
| 缺陷 | 是 | 需要缺陷匹配和处置建议 |
| 未知意图 | 否 | 直接输出固定引导回答 |

这体现了一个架构取舍：意图识别只决定入口类型，真正的工具选择、prompt 构建、流式生成都交给 UniQA。

在工具配置中，系统定义了多种工具：

| 工具 | 作用 |
| --- | --- |
| `knowledge_tool` | 变压器知识、标准、技术原理和通用方法说明 |
| `test_data_tool` | 查询指定变压器历史试验记录和统计信息 |
| `test_result_analysis_tool` | 对用户提供的试验数值做解读 |
| `hidden_risk_analysis_tool` | 分析变压器隐患风险 |
| `main_defect_analysis_tool` | 分析主缺陷 |
| `case_match_tool` | 匹配相似故障案例 |

这相当于把系统拆成两层：

1. **意图识别层**：判断用户问题应该进入哪类业务语义。
2. **工具编排层**：根据语义和上下文选择具体工具、构造 prompt、生成答案。

这样的分层比“一个 prompt 里既分类又查工具又回答”更稳定，也更容易调试。

## 九、SSE 事件：让前端知道系统正在做什么

TransformerDiag 的意图识别接口是流式接口。它不只是最后返回一个答案，而是会通过 SSE 推送多个阶段事件。

典型事件包括：

| 事件 | 作用 |
| --- | --- |
| `intent_recognition` | 告诉前端识别到的意图、置信度和下一步动作 |
| `test_qa_stream_start` | UniQA 或兜底问答开始输出 |
| `test_qa_stream_chunk` | LLM 流式输出片段 |
| `test_qa_result` | 最终回答结果 |
| `error` | UniQA 主流程异常时触发兜底 |

这对用户体验很重要。电力诊断和试验分析往往需要调用多个工具，如果前端只等待最终 HTTP 响应，用户会觉得系统卡住。SSE 能把“正在识别意图”“正在调用知识工具”“正在生成回答”这些过程显性化。

## 十、失败语义：工程系统必须能降级，而不是直接断流

项目文档里专门列了失败处理语义，这是它比较成熟的地方。

| 环节 | 失败策略 | 设计意义 |
| --- | --- | --- |
| `ctxopt /api/query` | 记录失败，继续使用当前问题 | 记忆增强不可用不影响基础问答 |
| LLM 意图识别 | 异常时降级为统一问答 | 避免入口直接失败 |
| 工具选择 LLM | 解析失败按空工具计划处理 | 防止工具计划格式错误拖垮回答 |
| `knowledge_tool` | 标记服务不可用，不作为事实依据 | 外部 RAG 失败不污染最终结论 |
| DST/Context 缺失 | 尝试补建，失败则走兜底问答 | 保持会话链路不断 |
| UniQA error 或无最终结果 | 切换 resilient fallback | 保证客户端收到 `test_qa_result` |
| PG 更新回答 | 记录错误 | 不影响当前响应，但需运维关注 |
| `ctxopt /api/store` | 后台异步写入 | 不阻塞 SSE 最终事件 |

这里最关键的是 `_handle_resilient_qa_fallback()`。当 UniQA 主流程异常、Context 缺失、事件映射失败或最终结果没输出时，系统会构造一个最小问答 prompt，继续通过 LLM 流式输出，并最终发送 `test_qa_result`。

这就是生产系统和 demo 系统的差别：demo 只要正常路径跑通；生产系统必须回答“任何一个外部服务挂了怎么办”。

## 十一、可观测性：`INTENT_CHAIN_TRACE` 是排障主线

TransformerDiag 在意图识别链路里使用结构化日志 `INTENT_CHAIN_TRACE`。它记录的字段非常贴近排障需求：

| 字段 | 含义 |
| --- | --- |
| `stage` | 当前链路阶段，如 `intent_prepared`、`ctxopt_store_scheduled`、`resilient_qa_fallback` |
| `raw_query` | 用户原始问题 |
| `rewritten_query` | ctxopt 改写后的当前问题 |
| `intent_input_has_history` | 意图识别输入是否携带 PG 历史 |
| `pg_context_count` | 用于辅助识别的历史轮数 |
| `tool_selection_result` | 工具选择结果 |
| `knowledge_api_called` | 是否调用知识工具 |
| `ctxopt_store_scheduled` | 是否调度后台记忆写入 |
| `fallback_reason` | 进入兜底流的原因 |

这类日志对意图识别尤其重要。因为很多线上问题不是“模型不会回答”，而是：

1. 当前问题被错误改写了。
2. 历史上下文污染了工具输入。
3. 设备 ID 继承错了。
4. 意图被硬模式误吸收。
5. 工具失败后仍被当作事实使用。

有了结构化链路日志，就可以从用户原文一路追到最终回答。

## 十二、我对这套设计的评价

TransformerDiag 的意图识别有几个很值得借鉴的点。

第一，**它把行业风险放进了意图边界**。试验结果解读和缺陷隐患都不是“看起来像”就进入，而是要求满足硬条件。这对电力这种高责任场景很关键。

第二，**它没有迷信单一 LLM 分类**。系统同时使用台账映射、上下文继承、ctxopt 改写、PG 历史、硬规则 prompt 和槽位管理。LLM 是语义判断器，但不是整个系统的唯一支点。

第三，**它把当前问题和历史上下文隔离得比较清楚**。这能避免多轮对话里最常见的上下文污染问题。

第四，**它有完整兜底意识**。ctxopt、RAG、UniQA、PG 写入任何一个环节失败，都有不同级别的处理策略，不会让前端只收到一个 error。

当然，这套系统也还有继续优化的空间：

| 方向 | 可优化点 |
| --- | --- |
| 评测集 | 建立覆盖试验解读、缺陷隐患、统一问答、未知意图的边界样本集 |
| 可解释性 | 把硬模式缺失要素显式返回给前端或日志 |
| 工具路由 | 将工具选择结果、工具失败原因纳入统一观测面板 |
| 安全策略 | 对工程处置建议增加风险等级、证据来源和人工复核提示 |
| Prompt 版本化 | 对 `intent_prompt.md` 做版本号、回归测试和灰度发布 |

## 十三、总结：意图识别是电力智能体的“调度室”

在 TransformerDiag 里，意图识别不是一个小模块，而是整个电力智能体的调度室。

它决定：

1. 用户是在问知识，还是要进入工程分析。
2. 当前问题是否满足硬模式参数。
3. 是否需要继承上一轮设备和上下文。
4. 是否调用 UniQA 和专业工具。
5. 工具失败时如何保证回答链路不断。
6. 会话历史如何写回，为下一轮追问服务。

这类设计给行业智能体一个很好的启发：真正可靠的智能体，不是把所有问题都交给一个大模型自由发挥，而是要在入口处建立清晰的语义边界、状态边界和风险边界。

TransformerDiag 的意图识别链路，本质上就是把“用户一句话”变成“可执行、可追踪、可降级的专业任务”的过程。
