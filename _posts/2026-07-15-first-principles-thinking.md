# 第一性原理：不是把问题想复杂，而是把假设拆干净

“第一性原理”经常被说成一种很高级的思考方式，但它真正有用的地方并不玄。它不是要求人每件事都从物理学开始推导，也不是为了显得反常识，而是要求我们在关键问题上先停下来问一句：

> 这个结论到底来自事实、约束和目标，还是来自习惯、类比和未经验证的假设？

## 一、什么是第一性原理

第一性原理的核心，是把问题拆到最基础、最不能再继续解释的事实和约束，然后从这些基础重新构造答案。

它和“类比思维”最大的区别在于：

- 类比思维问：别人是怎么做的？过去是怎么做的？
- 第一性原理问：事情本身必须满足什么？哪些条件真的不能变？

类比思维并不差。大多数日常决策都应该靠经验、范式和最佳实践，因为这样效率高。但当你面对的问题足够重要、足够复杂，或者行业默认答案已经明显失效时，第一性原理才值得启动。

一个简洁定义：

> 第一性原理不是拒绝经验，而是把经验拆开，区分其中的事实、约束、假设和惯例。

---

## 二、类比思维和第一性原理的差别

| 维度 | 类比思维 | 第一性原理 |
| --- | --- | --- |
| 起点 | 过去经验、行业惯例、竞品做法 | 目标、事实、约束、因果关系 |
| 典型问题 | “别人怎么做？” | “这件事本质上需要什么？” |
| 优势 | 快、稳、低成本 | 能突破惯性，适合创新和重构 |
| 风险 | 复制旧答案，继承旧假设 | 成本高，容易过度推导 |
| 适合场景 | 常规执行、成熟流程、低风险决策 | 高杠杆决策、系统设计、创新、瓶颈问题 |
| 判断标准 | 是否符合已有范式 | 是否满足基本事实和约束 |

真正成熟的思考方式不是二选一，而是知道什么时候用哪一个。

日常问题用类比，关键问题用第一性原理。否则你会把简单问题想复杂，也会把复杂问题做成复制粘贴。

---

## 三、第一性原理的四步法

### 1. 明确目标

很多人一上来就问“怎么做”，但第一性原理要求先问“为了什么”。

例如：

```text
问题：我们要不要做一个新的知识库系统？
普通问法：别人都用了什么知识库？
第一性原理问法：我们真正要降低的是检索成本、协作成本，还是知识复用成本？
```

目标不同，方案完全不同：

- 如果目标是降低检索成本，可能需要搜索和标签体系。
- 如果目标是降低协作成本，可能需要流程和权限设计。
- 如果目标是提升知识复用，可能需要模板、沉淀机制和内容质量治理。

没有目标，后面的“拆解”都会变成空转。

### 2. 拆解变量

把问题拆成基本变量，区分哪些是真事实，哪些是假设。

可以用一个简单表格：

| 类型 | 问法 | 示例 |
| --- | --- | --- |
| 事实 | 已经确定什么？ | 用户每天平均产生 200 条咨询 |
| 约束 | 什么不能违反？ | 响应时间不能超过 2 秒 |
| 假设 | 我们以为但未验证什么？ | 用户愿意主动填写标签 |
| 惯例 | 只是过去这样做吗？ | 所有知识必须按部门分类 |
| 目标 | 最终优化什么？ | 减少重复问答和人工转接 |

这一步的关键不是拆得多，而是拆得干净。

### 3. 找到硬约束

硬约束是不能靠想象绕过去的东西，比如物理限制、法律限制、预算限制、时间限制、组织能力限制。

很多创新不是因为“想得更大胆”，而是因为发现了过去被当成硬约束的东西其实只是软约束。

例如：

| 看似硬约束 | 拆开后可能发现 |
| --- | --- |
| 电池一定很贵 | 贵的是成品供应链，不一定是材料本身 |
| 用户一定不愿意学习新工具 | 用户不愿意学的是无收益的新工具 |
| 模型必须回答所有问题 | 高风险问题可以追问、拒答或转人工 |
| 系统必须全自动 | 关键环节可以人机协同 |

第一性原理不是无视约束，而是重新判断哪些约束是真的。

### 4. 重新构造方案

当你拆出事实、目标和约束后，就不要急着回到原来的答案。要问：

```text
如果不沿用旧方案，我能不能从这些基本单元重新组合出一个更好的方案？
```

这一步才是创新发生的地方。

## 四、几个典型案例

### 1. 电池成本：从成品价格回到材料成本

一个常被引用的例子是电池成本。传统类比思维可能是：电池组市场价很贵，所以电动车很难便宜。

第一性原理会继续拆：

- 电池由哪些材料组成？
- 这些材料的市场价格是多少？
- 加工、封装、供应链、规模化各占多少成本？
- 哪些成本会随规模下降？

这个拆解会把“电池天然昂贵”改写成一个更具体的问题：哪些环节真的贵，哪些只是供应链和规模问题？

### 2. 产品设计：从功能列表回到用户任务

很多产品迭代会陷入竞品类比：竞品有 A 功能，我们也要有；竞品有仪表盘，我们也做仪表盘。

第一性原理会问：用户真正要完成的任务是什么？

| 类比式问题 | 第一性原理问题 |
| --- | --- |
| 竞品有哪些功能？ | 用户要完成什么任务？ |
| 这个按钮放哪里？ | 用户下一步需要做什么决策？ |
| 要不要做报表？ | 用户是否需要比较、追踪或解释？ |
| 要不要加 AI？ | AI 能减少哪一步认知或操作成本？ |

如果回到用户任务，很多功能会变得不必要，另一些隐藏需求会浮现出来。

### 3. AI 系统：从“模型更大”回到失败链路

做 AI 产品时，很容易把问题归因到“模型不够强”。但第一性原理会把链路拆开：

```text
用户输入 -> 意图识别 -> 检索/工具调用 -> 上下文组织 -> 推理生成 -> 结构化输出 -> 校验与反馈
```

如果系统失败，可能不是模型本身弱，而是：

- 用户意图没有识别对。
- 检索证据不相关。
- 证据找到了但没被使用。
- prompt 把优先级写乱了。
- 输出没有校验。
- 业务流程本身没有定义清楚。

这时盲目换大模型只是类比式升级。第一性原理要求先找到真正瓶颈。

## 五、第一性原理不是万能药

第一性原理很有用，但也容易被误用。

| 误用方式 | 问题 | 更好的做法 |
| --- | --- | --- |
| 每件小事都从零推导 | 成本太高，效率低 | 常规问题用经验，关键问题再深挖 |
| 把常识全盘否定 | 容易重复踩坑 | 先理解经验为何成立，再判断是否过期 |
| 只拆解不验证 | 变成纸面推理 | 用小实验验证关键假设 |
| 只追求新方案 | 忽略执行成本 | 同时评估资源、时间和组织能力 |
| 把口号当方法 | 看似深刻，无法落地 | 输出变量、约束、假设和实验计划 |

第一性原理不是“我觉得别人都错了”，而是“我能证明哪些假设不必成立”。

## 六、如何在工作中使用第一性原理

可以用下面这张清单。

### 第一性原理检查清单

| 步骤 | 问题 | 输出物 |
| --- | --- | --- |
| 1. 定义目标 | 我们到底要优化什么？ | 一个清晰目标指标 |
| 2. 写出默认答案 | 行业惯例或当前方案是什么？ | 当前方案图 |
| 3. 标出假设 | 这个方案依赖哪些前提？ | 假设列表 |
| 4. 区分约束 | 哪些是真约束，哪些只是习惯？ | 约束分层表 |
| 5. 拆基本单元 | 问题由哪些变量构成？ | 变量表 |
| 6. 重新组合 | 有没有不同组合方式？ | 2-3 个候选方案 |
| 7. 设计实验 | 哪个假设最值得验证？ | 最小实验 |
| 8. 回到现实 | 成本、风险、收益是否成立？ | 决策建议 |

注意最后一步：回到现实。

第一性原理不是让你永远停留在抽象层，而是为了更准确地行动。

这个模板的价值在于，它逼迫你把“我觉得”变成“我知道什么、我假设什么、我要验证什么”。

## 八、第一性原理和创新的关系

创新不是凭空想象，而是重新排列基本单元。

很多所谓创新，本质上来自三种动作：

| 动作 | 含义 | 例子 |
| --- | --- | --- |
| 去掉假约束 | 发现某个限制并非必要 | 服务不一定必须人工交付 |
| 替换基本单元 | 用新的技术或资源替代旧单元 | 用模型检索替代人工查找 |
| 重新定义目标 | 不再优化旧指标 | 不追求更多功能，而追求更少操作 |

第一性原理能帮助创新，是因为它把问题从“已有答案空间”里拿出来，重新回到“约束和目标空间”。

---

## 九、总结

第一性原理不是一种装饰性概念，而是一种工程化的思考工具。

它的核心流程很简单：

```text
明确目标
-> 拆解变量
-> 区分事实、假设、惯例和约束
-> 回到基本单元
-> 重新构造方案
-> 用实验验证
```

最重要的是：不要把第一性原理理解成反经验。经验是压缩过的历史，通常很有价值。但当环境变化、问题复杂、默认答案失效时，我们就需要把经验解压，看看里面到底有多少事实，多少假设，多少只是时代留下的惯性。

真正有用的第一性原理，不是让你显得聪明，而是让你更少被默认答案困住。
