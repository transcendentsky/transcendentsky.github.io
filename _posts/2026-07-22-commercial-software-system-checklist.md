> 一句话总结：个人 demo 证明“这个想法能不能跑起来”，商业软件系统要证明“这个系统能不能长期、稳定、安全、可维护地服务真实用户”。两者的差异不在代码行数，而在需求边界、质量标准、测试体系、发布流程、运维能力和风险责任。

![商业软件系统关注面](/assets/images/commercial-software-system/commercial-system-scope.svg)

<!--more-->

很多项目最开始都是 demo。

个人 demo 的目标通常很直接：用最短时间证明一个想法能不能成立。只要核心路径能跑通，UI 差一点、异常处理少一点、测试不完整，都可以接受。

但商业软件系统不一样。

商业系统面对的是真实用户、真实数据、真实付费、真实故障和真实责任。它不能只在开发者机器上跑通，也不能只覆盖 happy path。它需要在长周期里持续交付、持续运行、持续迭代。

因此，从 demo 到商业系统，工程关注点会发生根本变化：

| 维度 | 个人 Demo | 商业软件系统 |
| --- | --- | --- |
| 目标 | 证明想法 | 支撑业务 |
| 用户 | 自己或少数人 | 真实客户、运营、客服、管理员 |
| 数据 | 示例数据 | 真实数据、敏感数据、历史数据 |
| 故障 | 手动修 | 要监控、告警、恢复、复盘 |
| 测试 | 跑通主流程 | 覆盖功能、性能、安全、兼容、回归 |
| 发布 | 手动部署 | CI/CD、灰度、回滚 |
| 权限 | 简单账号 | RBAC、审计、最小权限 |
| 成本 | 不敏感 | 要预算、容量、成本优化 |
| 合规 | 很少考虑 | 隐私、许可、日志、数据保留 |

本文按工程实践拆解：商业软件系统应该考虑哪些方面、做哪些测试，以及相比个人 demo 到底多了什么。

## 一、需求不再只是“能用”，而是“可验收”

个人 demo 常见写法是：

```text
做一个 AI 文档问答工具。
```

商业系统必须把需求拆清楚：

```text
给企业内部员工使用的文档问答系统。
支持 5000 份文档，日活 300 人。
回答必须给出引用来源。
P95 响应时间不超过 5 秒。
无权限文档不能被检索。
管理员可以更新知识库。
每次问答要记录审计日志。
```

商业系统的需求至少要明确：

| 问题 | 示例 |
| --- | --- |
| 用户是谁 | 普通用户、管理员、运营、客服、审计员 |
| 核心场景是什么 | 查询、下单、审批、分析、配置 |
| 成功标准是什么 | 准确率、响应时间、转化率、错误率 |
| 失败怎么处理 | 重试、降级、人工介入、回滚 |
| 边界在哪里 | 不支持哪些输入、哪些角色不能做什么 |
| 数据从哪里来 | 用户输入、第三方 API、历史数据库 |
| 验收标准是什么 | 可执行测试用例，而不是口头描述 |

一个需求如果不能验收，就不能稳定交付。

## 二、用户角色和权限要提前设计

demo 通常只有一个用户：开发者自己。

商业系统至少要考虑：

1. 匿名用户。
2. 普通登录用户。
3. 付费用户。
4. 管理员。
5. 运营人员。
6. 审计人员。
7. 第三方系统调用方。

权限模型要回答：

| 问题 | 说明 |
| --- | --- |
| 谁能看什么数据 | 数据级权限 |
| 谁能执行什么操作 | 操作级权限 |
| 谁能配置系统 | 管理权限 |
| 谁能导出数据 | 高风险权限 |
| 谁能查看日志 | 审计权限 |
| 权限变更是否留痕 | 合规要求 |

最少要做：

1. RBAC 或 ABAC 权限设计。
2. 登录态和 token 过期机制。
3. 管理端权限隔离。
4. 越权访问测试。
5. 敏感操作审计。

不要等系统上线后再补权限。权限是架构，不是补丁。

## 三、数据模型要考虑生命周期

demo 可以用几条 mock 数据。

商业系统要考虑数据从创建到删除的整个生命周期：

| 阶段 | 要考虑什么 |
| --- | --- |
| 创建 | 字段校验、唯一性、幂等 |
| 更新 | 并发冲突、版本号、审计 |
| 查询 | 索引、分页、权限过滤 |
| 删除 | 软删除、恢复、级联影响 |
| 归档 | 历史数据、冷热分层 |
| 备份 | RPO、RTO、恢复演练 |
| 合规 | 数据保留、脱敏、删除请求 |

常见问题：

1. 字段没有唯一约束，导致重复数据。
2. 删除是真删，后来无法恢复。
3. 查询不分页，数据一多就慢。
4. 没有数据版本，更新冲突无法处理。
5. 日志里写入明文敏感信息。
6. 测试环境使用生产数据却没脱敏。

商业系统的数据模型必须能承受增长和错误。

## 四、架构要考虑扩展、隔离和故障

个人 demo 常见架构：

```text
前端 + 后端 + SQLite + 一个脚本
```

商业系统要考虑：

1. 服务边界。
2. 数据库边界。
3. 缓存。
4. 消息队列。
5. 文件存储。
6. 第三方接口。
7. 后台任务。
8. 配置中心。
9. 监控和日志。

不是说一开始就要微服务。很多商业系统初期用单体架构更合理。但即使用单体，也要把模块边界设计清楚：

| 模块 | 要求 |
| --- | --- |
| 用户模块 | 鉴权、权限、审计 |
| 业务模块 | 核心规则清晰 |
| 数据模块 | 数据一致性 |
| 支付模块 | 幂等、对账、异常处理 |
| 通知模块 | 重试、去重、失败记录 |
| 文件模块 | 权限、存储、病毒扫描 |
| AI 模块 | 提示词版本、评测、降级 |

商业系统的架构目标不是看起来高级，而是让系统在变化和故障下仍然可控。

## 五、异常处理和边界条件

demo 常见问题是只处理正常路径。

商业系统必须处理：

| 类型 | 示例 |
| --- | --- |
| 输入异常 | 空值、超长、非法格式、重复提交 |
| 状态异常 | 已取消订单再次支付 |
| 并发异常 | 两个人同时修改同一条记录 |
| 依赖异常 | 第三方 API 超时 |
| 数据异常 | 历史脏数据、缺字段 |
| 权限异常 | 用户访问不属于自己的资源 |
| 网络异常 | 请求重试导致重复写入 |
| 时间异常 | 跨时区、夏令时、月底结算 |

关键策略：

1. 输入校验放在边界。
2. 核心业务规则放在服务层。
3. 写操作要考虑幂等。
4. 外部调用要有超时和重试。
5. 失败要有明确错误码。
6. 用户可理解的错误信息和内部日志分开。

## 六、测试体系：商业系统不能只测 happy path

商业软件的测试目标不是证明代码“能跑”，而是尽可能在上线前暴露风险。

![商业系统测试矩阵](/assets/images/commercial-software-system/testing-matrix.svg)

测试至少分几类：

| 测试类型 | 目标 | 示例 |
| --- | --- | --- |
| 单元测试 | 验证函数和业务规则 | 金额计算、权限判断、状态流转 |
| 集成测试 | 验证模块协作 | API + DB + Cache |
| E2E 测试 | 验证真实用户路径 | 注册、下单、支付、退款 |
| 回归测试 | 防止旧功能被破坏 | 历史 bug 用例 |
| 性能测试 | 验证容量和延迟 | QPS、P95、P99 |
| 安全测试 | 发现安全漏洞 | 越权、注入、XSS、CSRF |
| 兼容性测试 | 覆盖环境差异 | 浏览器、移动端、系统版本 |
| 可用性测试 | 验证用户是否会用 | 表单、流程、文案 |
| 故障演练 | 验证系统能恢复 | DB 慢、缓存挂、第三方超时 |

一个成熟系统不一定测试数量最多，但测试结构必须合理。

## 七、单元测试：保护业务规则

单元测试适合覆盖稳定、明确的业务规则。

典型场景：

1. 金额计算。
2. 折扣规则。
3. 权限判断。
4. 状态机流转。
5. 输入解析。
6. 数据转换。
7. 风控规则。

示例测试点：

```text
订单状态为 paid 时，不能再次支付。
普通用户不能导出全量客户数据。
优惠券过期后不能使用。
退款金额不能超过已支付金额。
```

好的单元测试有三个特点：

1. 不依赖外部服务。
2. 运行快。
3. 失败原因明确。

不要为了覆盖率写无意义测试。测试应该锁定业务风险。

## 八、集成测试：验证真实依赖

商业系统很多 bug 不在单个函数里，而在模块连接处：

1. API 参数和数据库字段不一致。
2. Redis 缓存没有失效。
3. 消息队列重复消费。
4. 第三方 API 返回格式变化。
5. 数据库事务边界错误。

集成测试应该覆盖：

| 依赖 | 测试重点 |
| --- | --- |
| 数据库 | 事务、唯一约束、迁移 |
| 缓存 | 失效、穿透、雪崩 |
| MQ | 重试、幂等、死信 |
| 文件存储 | 上传、下载、权限 |
| 第三方 API | 超时、错误码、重试 |
| AI 服务 | 超时、限流、输出格式 |

集成测试可以慢一点，但必须真实。

## 九、E2E 测试：验证用户路径

E2E 测试用于覆盖关键业务路径。

不要把所有场景都做成 E2E。E2E 成本高、慢、容易不稳定。应该重点覆盖：

1. 登录和权限。
2. 创建核心资源。
3. 支付或提交。
4. 审批流。
5. 文件上传下载。
6. 管理端关键操作。
7. AI 系统的典型问答或任务执行。

E2E 测试的价值是发现“单个模块都没问题，但用户流程走不通”的问题。

## 十、性能和容量测试

demo 只要一个人用起来不卡就行。

商业系统要回答：

1. 同时多少用户？
2. 峰值 QPS 多少？
3. P95/P99 延迟是多少？
4. 数据增长后查询是否变慢？
5. 队列积压后多久恢复？
6. 第三方接口慢了系统会怎样？
7. 成本随流量怎么增长？

性能测试至少包括：

| 类型 | 目标 |
| --- | --- |
| 基准测试 | 单服务基础性能 |
| 负载测试 | 正常高峰流量 |
| 压力测试 | 找系统极限 |
| 稳定性测试 | 长时间运行 |
| 容量测试 | 数据量增长后的表现 |
| 突刺测试 | 瞬时流量冲击 |

关键指标：

1. QPS。
2. P50/P95/P99 latency。
3. 错误率。
4. CPU/内存/GPU/磁盘。
5. DB 慢查询。
6. 队列积压。
7. 成本。

性能测试不能只测空系统。要准备接近真实的数据规模。

## 十一、安全测试

商业系统必须把安全作为基本工程要求。

至少要测：

| 类别 | 测试点 |
| --- | --- |
| 认证 | 登录、token 过期、会话固定 |
| 授权 | 水平越权、垂直越权 |
| 输入安全 | SQL 注入、命令注入、XSS |
| 文件安全 | 文件类型、大小、病毒、路径穿越 |
| API 安全 | 重放、签名、限流 |
| 数据安全 | 明文密码、敏感日志、备份泄露 |
| 依赖安全 | 漏洞库、许可证风险 |
| AI 安全 | Prompt injection、越权检索、敏感输出 |

对于 AI 系统，还要额外测试：

1. 用户能否通过 prompt 获取无权限数据。
2. RAG 是否会检索到不该看的文档。
3. 模型是否会泄露 system prompt。
4. 工具调用是否被恶意输入诱导。
5. 输出是否包含敏感信息。

AI 系统的安全边界不能只靠 prompt，要靠权限、检索过滤、工具白名单和审计。

## 十二、数据迁移和兼容性测试

商业系统上线后一定会改数据库。

每次 schema 变更都要考虑：

1. 老数据能不能迁移。
2. 回滚后数据是否兼容。
3. 新旧版本服务能否短时间共存。
4. 迁移脚本能否重复执行。
5. 大表迁移是否锁表。
6. 是否需要灰度字段。

测试方法：

1. 用生产规模抽样数据演练。
2. 迁移前备份。
3. 迁移后校验行数和关键字段。
4. 演练回滚路径。
5. 监控慢查询和锁等待。

数据库迁移是商业系统里最容易造成线上事故的环节之一。

## 十三、可观测性：日志、指标、链路追踪

demo 出问题时，开发者可以直接调试。

商业系统出问题时，用户只会看到“不可用”。

必须建立可观测性：

| 类型 | 看什么 |
| --- | --- |
| 日志 | 请求、错误、关键业务事件 |
| 指标 | QPS、延迟、错误率、资源使用 |
| 链路追踪 | 一次请求跨服务耗时 |
| 审计日志 | 谁在什么时候做了什么 |
| 业务指标 | 转化率、成功率、订单量 |
| AI 指标 | token 用量、失败率、bad cases |

日志要注意：

1. 不打印密码、token、身份证、银行卡。
2. 错误日志要包含 request id。
3. 用户错误和系统错误分开。
4. 关键业务操作要留审计日志。
5. 日志保留周期要明确。

可观测性的目标是让问题定位从“猜”变成“查”。

## 十四、发布、灰度和回滚

个人 demo 可以直接覆盖部署。

商业系统必须考虑发布风险。

推荐流程：

1. 代码评审。
2. 自动化测试。
3. 构建制品。
4. 部署测试环境。
5. 回归测试。
6. 灰度发布。
7. 观察指标。
8. 全量发布。
9. 保留回滚方案。

![商业系统发布反馈闭环](/assets/images/commercial-software-system/release-feedback-loop.svg)

每次上线前至少回答：

| 问题 | 要求 |
| --- | --- |
| 改了什么 | 变更说明 |
| 影响谁 | 影响范围 |
| 怎么验证 | 测试报告 |
| 怎么观测 | 指标和日志 |
| 出问题怎么办 | 回滚方案 |
| 是否需要数据迁移 | 迁移脚本和回滚 |
| 是否需要通知用户 | 运营/客服准备 |

没有回滚方案的发布，是不完整的发布。

## 十五、运维和故障处理

商业系统要假设故障一定会发生。

需要准备：

1. 服务健康检查。
2. 告警规则。
3. 值班机制。
4. 故障分级。
5. 应急预案。
6. 数据备份和恢复。
7. 故障复盘。

常见故障演练：

| 故障 | 要验证 |
| --- | --- |
| 数据库变慢 | 系统是否降级 |
| Redis 不可用 | 是否能退化到 DB |
| MQ 积压 | 是否能扩容消费者 |
| 第三方 API 超时 | 是否重试和熔断 |
| AI 服务不可用 | 是否切换模型或降级 |
| 磁盘满 | 是否告警 |
| 新版本异常 | 是否能快速回滚 |

商业系统不能靠“工程师刚好在线”维持稳定。

## 十六、成本和容量

demo 几乎不考虑成本。

商业系统必须考虑：

1. 服务器成本。
2. 数据库成本。
3. 存储成本。
4. 带宽成本。
5. 第三方 API 成本。
6. AI token 成本。
7. 日志和监控成本。

尤其是 AI 系统，要记录：

| 指标 | 作用 |
| --- | --- |
| 每请求 token 数 | 控制模型成本 |
| 每用户调用次数 | 做限额 |
| RAG 上下文长度 | 控制输入成本 |
| 缓存命中率 | 降低重复调用 |
| 模型切换策略 | 大小模型分层 |

商业系统的性能优化最终都要落到成本和用户体验之间的平衡。

## 十七、文档和交接

个人 demo 可以没有文档。

商业系统至少要有：

1. 架构文档。
2. API 文档。
3. 数据库 schema 文档。
4. 部署文档。
5. 运维手册。
6. 故障处理手册。
7. 权限说明。
8. 测试说明。
9. 版本变更记录。

文档不是形式主义。文档解决的是团队协作和长期维护问题。

判断文档是否合格的标准：

> 新人能不能在不打扰核心开发者的情况下，把系统跑起来、改一个小需求、定位一个常见问题。

## 十八、相比个人 demo，商业系统新增的核心测试清单

可以用下面清单做项目评审。

| 测试项 | 是否必须 |
| --- | --- |
| 核心业务单元测试 | 必须 |
| API 集成测试 | 必须 |
| 数据库迁移测试 | 必须 |
| 权限越权测试 | 必须 |
| 关键路径 E2E 测试 | 必须 |
| 回归测试 | 必须 |
| 性能压测 | 必须 |
| 安全扫描 | 必须 |
| 兼容性测试 | 视产品而定 |
| 可用性测试 | 面向用户产品必须 |
| 灰度验证 | 必须 |
| 回滚演练 | 必须 |
| 备份恢复演练 | 关键系统必须 |
| AI bad case 测试 | AI 系统必须 |
| Prompt injection 测试 | AI 系统必须 |

商业系统不是测试越多越好，而是关键风险必须被覆盖。

## 十九、一个实用的商业化上线门槛

在系统进入正式商业使用前，至少满足：

1. 核心需求有明确验收标准。
2. 核心路径有自动化测试。
3. 权限和数据边界经过测试。
4. 关键 API 有集成测试。
5. 数据库迁移有演练。
6. 性能达到预期容量。
7. 安全扫描无高危漏洞。
8. 监控、日志、告警可用。
9. 发布支持灰度和回滚。
10. 有故障处理手册。
11. 有备份恢复策略。
12. 成本模型可估算。
13. 用户反馈渠道明确。
14. 关键文档可交接。

如果这些都没有，系统可能可以 demo，但还不应该称为商业系统。

## 二十、总结

个人 demo 和商业软件系统最大的区别，不是功能多少，而是责任不同。

个人 demo 关注：

1. 想法是否成立。
2. 核心流程是否跑通。
3. 技术方案是否可行。

商业系统关注：

1. 需求是否可验收。
2. 数据是否安全。
3. 权限是否正确。
4. 系统是否稳定。
5. 测试是否覆盖关键风险。
6. 发布是否可回滚。
7. 故障是否可定位。
8. 成本是否可控。
9. 团队是否可维护。
10. 用户是否能持续获得价值。

从 demo 到商业系统，本质上是从“证明可能性”走向“承担确定性”。

这也是软件工程真正的分水岭。
