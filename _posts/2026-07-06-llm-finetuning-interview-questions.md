# 大模型微调面试题整理：从 SFT、LoRA 到对齐与部署

这篇整理面向大模型微调相关岗位面试，覆盖基础概念、数据构造、SFT、LoRA/QLoRA、偏好对齐、评估、部署、排障和工程落地。

如果只准备一条主线，可以记住这句话：

> 大模型微调不是简单“继续训练一下”，而是围绕业务目标，系统设计数据、训练方法、评估指标、部署约束和持续迭代机制。

<!--more-->

---

## 一、基础概念

### 1. 什么是大模型微调？它和预训练有什么区别？

**答案：**

预训练是在大规模通用语料上学习语言、知识和基础推理能力，目标通常是预测下一个 token。微调是在已有基础模型上，用更小、更聚焦的数据，让模型适应特定任务、领域、格式或偏好。

区别主要有四点：

- 数据规模：预训练通常是 TB 级甚至更大，微调通常是千到百万级样本。
- 目标：预训练学习通用能力，微调让模型更符合具体任务。
- 成本：预训练成本极高，微调成本相对可控。
- 风险：微调容易带来过拟合、灾难性遗忘和格式偏移。

---

### 2. 大模型微调主要解决哪些问题？

**答案：**

常见目标包括：

- 让模型学会特定输出格式，例如 JSON、SQL、报告模板。
- 增强垂直领域能力，例如医疗、法律、金融、工业知识。
- 提升特定任务表现，例如分类、抽取、问答、代码生成。
- 对齐企业风格，例如客服话术、品牌语气、合规表达。
- 学会工具调用、函数调用、多轮任务流程。

但微调不是万能的。如果问题是知识实时性、私有知识检索、事实更新，RAG 往往比微调更合适。

---

### 3. 微调、RAG、Prompt Engineering 分别适合什么场景？

**答案：**

Prompt Engineering 适合快速验证、任务简单、格式要求不复杂的场景。RAG 适合依赖外部知识、知识频繁变化、需要引用来源的场景。微调适合需要稳定行为模式、固定输出格式、领域表达习惯、任务能力内化的场景。

一个实用判断：

- 模型“不知道”：优先 RAG。
- 模型“知道但不会按你的方式做”：考虑微调。
- 模型“偶尔格式错”：先改 prompt 和解码约束，再考虑微调。
- 模型“要学一个稳定流程”：SFT 或工具调用微调更合适。

---

### 4. SFT 是什么？

**答案：**

SFT 是 Supervised Fine-Tuning，监督微调。它使用输入和理想输出组成的样本训练模型，让模型在给定指令时生成期望答案。

典型样本结构是：

```json
{
  "instruction": "请抽取合同中的甲方、乙方、金额和日期",
  "input": "合同正文...",
  "output": "{\"甲方\":\"...\",\"乙方\":\"...\"}"
}
```

SFT 的核心不是让模型背知识，而是让模型学习任务映射、输出风格、格式约束和工作流程。

---

### 5. Instruction Tuning 和 SFT 是同一个东西吗？

**答案：**

二者高度相关，但侧重点不同。

SFT 是训练方法，指用有监督样本继续训练模型。Instruction Tuning 是一种 SFT 场景，重点是让模型理解和遵循自然语言指令。

可以理解为：

```text
SFT 是方法
Instruction Tuning 是 SFT 的一种典型用途
```

---

### 6. 什么是全参数微调？

**答案：**

全参数微调是指更新模型所有参数。优点是表达能力强，对任务适配充分；缺点是显存和计算成本高，需要保存完整模型副本，也更容易发生灾难性遗忘。

全参数微调适合：

- 数据量足够大。
- 领域差异明显。
- 对效果要求极高。
- 有充足训练资源。

在多数企业落地场景中，LoRA、QLoRA 等参数高效微调方法更常见。

---

### 7. 什么是 PEFT？

**答案：**

PEFT 是 Parameter-Efficient Fine-Tuning，参数高效微调。它不更新全部参数，而是只训练少量新增参数或部分参数，从而降低显存、存储和训练成本。

常见 PEFT 方法包括：

- LoRA
- QLoRA
- Prefix Tuning
- Prompt Tuning
- Adapter Tuning

目前大模型微调里最常见的是 LoRA 和 QLoRA。

---

### 8. 为什么现在很多大模型微调都用 LoRA？

**答案：**

因为 LoRA 在效果、成本和工程便利性之间比较平衡。

它的优势包括：

- 只训练少量低秩矩阵，显存占用低。
- 可以快速训练多个任务 adapter。
- 原始模型权重保持不变，便于回滚和组合。
- 推理时可以合并权重，减少额外延迟。
- 对多数 SFT 场景效果足够好。

面试中可以补一句：LoRA 的价值不只是“省显存”，还在于让模型能力定制变得模块化。

---

### 9. 什么是灾难性遗忘？

**答案：**

灾难性遗忘是指模型在微调新任务后，原有通用能力或其他任务能力显著下降。例如一个通用聊天模型经过狭窄客服数据微调后，变得只会客服话术，开放问答能力下降。

常见缓解方法：

- 使用高质量、多样化数据。
- 控制学习率和训练轮数。
- 混入通用指令数据。
- 使用 PEFT 而不是全参数微调。
- 做微调前后的通用能力回归评估。

---

### 10. 什么时候不应该微调？

**答案：**

以下场景不建议优先微调：

- 只是补充新知识或私有知识，RAG 更合适。
- 数据质量很差，微调会放大噪声。
- 样本量极少，prompt 或 few-shot 可能更划算。
- 输出格式可以用规则、约束解码或后处理解决。
- 业务需求频繁变化，微调迭代成本太高。
- 没有可靠评估集，无法判断微调是否真的变好。

## 二、数据工程

### 11. 大模型微调数据通常长什么样？

**答案：**

常见格式包括：

- 单轮指令数据：instruction + output。
- 多轮对话数据：messages 列表。
- 任务样本：input + label 或 input + structured output。
- 偏好数据：prompt + chosen + rejected。
- 工具调用数据：用户请求 + tool call + tool result + final answer。

多轮对话格式通常类似：

```json
{
  "messages": [
    {"role": "system", "content": "你是一个严谨的医学助手"},
    {"role": "user", "content": "这个检查结果说明什么？"},
    {"role": "assistant", "content": "需要结合..."}
  ]
}
```

---

### 12. 微调数据质量比数量更重要吗？

**答案：**

通常是的。对于 SFT，几千条高质量、覆盖关键场景的数据，往往比几十万条低质量数据更有价值。

高质量数据的特征：

- 输入真实，接近线上分布。
- 答案准确，没有幻觉。
- 格式稳定，标注规范一致。
- 覆盖常见场景、边界场景和失败场景。
- 没有大量重复样本。
- 没有混入错误价值观或错误业务规则。

数据质量差时，模型会学到错误模式，而且微调后问题会变得更稳定、更难通过 prompt 修正。

---

### 13. 如何构造高质量 SFT 数据？

**答案：**

可以按以下流程：

1. 明确任务边界：模型到底要学什么，不学什么。
2. 收集真实输入：来自日志、业务文档、人工构造或专家案例。
3. 制定标注规范：格式、语气、字段、拒答规则、引用规则。
4. 人工标注或模型辅助生成初稿。
5. 专家审核关键样本。
6. 去重、清洗、脱敏、过滤低质量样本。
7. 划分训练集、验证集、测试集。
8. 做训练前基线测试。

---

### 14. 为什么训练集和测试集要去重？

**答案：**

如果测试集样本泄漏到训练集，评估结果会虚高。大模型很强的记忆能力会让它在测试时看起来表现很好，但真实泛化能力并没有提升。

去重包括：

- 完全重复去重。
- 近似重复去重。
- 同一问题不同写法的语义去重。
- 同一文档切片导致的上下文泄漏检查。

尤其在 RAG、文档问答和代码任务里，泄漏非常常见。

---

### 15. 数据里是否应该保留错误示例？

**答案：**

要看训练目标。

SFT 通常不直接训练错误答案，因为模型会模仿输出。但可以保留“用户错误输入”对应的正确处理方式，例如拒答、澄清、纠错。

偏好优化中可以使用错误示例作为 rejected answer，让模型学习哪个答案更差。

所以：

- SFT：输入可以是错误场景，输出必须是正确处理。
- DPO/RLHF：可以显式使用好坏答案对。

---

### 16. 如何处理长文档微调数据？

**答案：**

长文档任务要关注上下文长度和任务目标。

常见方法：

- 截断到模型可接受长度。
- 分段构造样本。
- 使用 sliding window。
- 用摘要或结构化信息替代全文。
- 长文档问答场景优先考虑 RAG。
- 如果模型支持长上下文，可以构造长上下文指令数据，但成本更高。

不要简单把超长文档全部塞进训练样本，否则训练成本高、噪声大，还可能让模型学不到关键任务。

---

### 17. 微调数据需要系统提示词吗？

**答案：**

如果线上推理会使用固定 system prompt，那么训练数据里最好保持一致。否则训练时和推理时分布不一致，模型可能表现不稳定。

常见做法：

- 把角色、边界、合规要求写在 system message。
- user message 放真实用户输入。
- assistant message 放理想输出。

但 system prompt 不应过长。过长会浪费上下文，也可能让模型过度依赖提示词。

---

### 18. 如何构造拒答数据？

**答案：**

拒答数据要让模型学会“不该回答时怎么回答”。

好的拒答样本包括：

- 明确不可回答原因。
- 给出安全替代建议。
- 不泄露敏感信息。
- 不输出无意义套话。

例如：

```text
用户：帮我写一个绕过验证码的脚本。
助手：我不能帮助绕过验证码或规避安全机制。如果你是在做合法测试，可以考虑...
```

拒答数据不能过多，否则模型会变得保守，正常问题也拒答。

---

### 19. 微调数据中如何做脱敏？

**答案：**

脱敏目标是避免模型记住或泄露个人信息、商业机密和敏感数据。

常见做法：

- 替换姓名、电话、地址、身份证、邮箱。
- 删除内部账号、密钥、URL、数据库连接串。
- 保留业务结构但替换实体值。
- 对医疗、金融、法律数据做合规审查。

脱敏要注意不要破坏任务本身。例如信息抽取任务需要保留字段结构，可以把真实值替换成虚拟值。

---

### 20. 合成数据能不能用于微调？

**答案：**

可以，但要谨慎。

合成数据的优点是成本低、可扩展、能覆盖边界场景。缺点是容易同质化、带入生成模型的幻觉和偏见。

好的做法：

- 用真实数据定义分布，再生成扩展样本。
- 用强模型生成，人工抽检。
- 加入多样化输入和反例。
- 用验证集确认合成数据是否真的提升效果。
- 避免让模型只学习另一个模型的口癖。

## 三、训练方法

### 21. SFT 的损失函数是什么？

**答案：**

通常是 next-token prediction 的交叉熵损失。给定输入和目标输出，模型学习在每个位置预测正确 token。

在对话微调中，通常只对 assistant 输出部分计算 loss，不对 user prompt 计算 loss。这样模型学习“如何回答”，而不是学习复述用户输入。

---

### 22. 为什么有时只对 assistant 部分计算 loss？

**答案：**

因为训练目标是让模型生成助手回复，而不是生成用户问题或 system prompt。

如果对整段对话都计算 loss，模型会浪费能力学习用户输入分布，还可能在推理时模仿用户说话。

所以常见做法是：

```text
system/user tokens: loss mask = 0
assistant tokens: loss mask = 1
```

---

### 23. 学习率设置过大会怎样？

**答案：**

学习率过大会导致：

- loss 不稳定甚至发散。
- 模型输出格式崩坏。
- 原有能力下降。
- 过快记住训练集噪声。
- 出现重复、乱码、过度自信等问题。

大模型微调通常使用较小学习率。LoRA 学习率可以比全参数微调略高，但也要通过验证集和小规模试跑确定。

---

### 24. epoch 越多越好吗？

**答案：**

不是。epoch 太多容易过拟合，尤其在数据量小、样本重复高时。

过拟合表现包括：

- 训练 loss 继续下降，验证集指标变差。
- 输出变得模板化。
- 对训练格式依赖很强。
- 换个问法效果明显下降。

常见做法是监控 validation loss、任务指标和人工评测，必要时 early stopping。

---

### 25. batch size 对微调有什么影响？

**答案：**

batch size 影响训练稳定性、显存占用和泛化。

- batch size 大：梯度更稳定，但显存占用高，可能泛化变差。
- batch size 小：噪声更大，显存压力低，但训练不稳定。

实际中常用 gradient accumulation 模拟更大的 effective batch size。

```text
effective batch size = per_device_batch_size × GPU 数量 × gradient_accumulation_steps
```

---

### 26. warmup 有什么作用？

**答案：**

warmup 是训练初期逐步增大学习率，避免一开始参数更新过猛。大模型微调中，warmup 可以提升训练稳定性，减少 early stage loss 抖动。

常见设置是总 steps 的 1% 到 5%，具体取决于数据规模和训练稳定性。

---

### 27. gradient checkpointing 是什么？

**答案：**

gradient checkpointing 是一种省显存技术。它不保存所有中间激活，而是在反向传播时重新计算部分激活。

优点：

- 显著降低显存占用。
- 可以训练更大模型或更长序列。

缺点：

- 训练速度变慢，因为需要重复计算。

---

### 28. mixed precision 训练有什么用？

**答案：**

混合精度使用 FP16 或 BF16 来减少显存占用、提升训练速度。

常见选择：

- FP16：显存省，但数值稳定性稍差。
- BF16：数值范围更大，更稳定，现代 GPU 常用。

大模型训练中，如果硬件支持，BF16 通常更稳。

---

### 29. 什么是梯度爆炸和梯度裁剪？

**答案：**

梯度爆炸是指梯度变得非常大，导致参数更新过度，训练不稳定。梯度裁剪是限制梯度范数，避免更新过猛。

在大模型微调中，如果出现 loss 突然变大、NaN、输出崩坏，可以检查学习率、数据异常和梯度裁剪设置。

---

### 30. 如何判断微调是否收敛？

**答案：**

不能只看训练 loss。更可靠的判断包括：

- 训练 loss 平稳下降。
- 验证 loss 不再明显下降。
- 核心任务指标提升。
- 人工评测质量提升。
- 输出格式稳定。
- 通用能力没有明显下降。

最终要看业务目标，而不是只看 loss。

---

## 四、LoRA 和 QLoRA

### 31. LoRA 的核心思想是什么？

**答案：**

LoRA 假设模型微调所需的参数更新可以用低秩矩阵近似。它冻结原模型权重，只训练两个小矩阵 A 和 B。

原本全参数更新是：

```text
W' = W + ΔW
```

LoRA 近似为：

```text
ΔW = B × A
```

其中 A 和 B 的秩很小，因此训练参数量大幅减少。

---

### 32. LoRA 的 r 是什么？

**答案：**

r 是低秩矩阵的 rank，控制 LoRA adapter 的表达能力。

- r 越大：可训练参数越多，表达能力更强，但显存和过拟合风险增加。
- r 越小：训练更轻量，但可能欠拟合。

常见取值包括 8、16、32、64。具体要根据模型规模、数据量和任务复杂度选择。

---

### 33. LoRA alpha 是什么？

**答案：**

LoRA alpha 是缩放系数，用来控制 LoRA 更新对原始权重的影响。常见形式是：

```text
output = W x + (alpha / r) × B A x
```

alpha 太小可能学不到足够变化，太大可能扰动原模型能力。常见经验是 alpha 取 r 的 1 到 2 倍，但要通过实验验证。

---

### 34. LoRA dropout 有什么作用？

**答案：**

LoRA dropout 用于防止 adapter 过拟合。它会在训练时随机丢弃部分 LoRA 分支输入。

数据量小、任务单一时适当 dropout 有帮助。数据量较大或任务需要强拟合时，dropout 可以设低甚至关闭。

---

### 35. LoRA 应该加在哪些层？

**答案：**

常见做法是在 attention projection 层加 LoRA，例如 q_proj、k_proj、v_proj、o_proj。也可以加到 MLP 层，例如 gate_proj、up_proj、down_proj。

经验上：

- 只加 attention：参数少，速度快，适合多数任务。
- attention + MLP：表达能力更强，成本更高。
- 任务复杂或领域差异大时，可以扩大 target modules。

---

### 36. LoRA 和 Adapter Tuning 有什么区别？

**答案：**

LoRA 在已有线性层上加入低秩更新，推理时可以合并进原权重。Adapter Tuning 通常在 Transformer 层中插入小型网络模块。

区别：

- LoRA 参数少，推理可合并，工程上常用。
- Adapter 更像新增网络层，结构更显式，但推理时通常保留额外模块。

---

### 37. QLoRA 是什么？

**答案：**

QLoRA 是在量化基础模型的同时训练 LoRA adapter 的方法。通常把基础模型量化到 4-bit，只训练 LoRA 参数，从而显著降低显存。

QLoRA 的关键点：

- 基础模型 4-bit 量化加载。
- LoRA adapter 用较高精度训练。
- 使用 NF4、double quantization 等技术降低损失。

它适合资源有限但想微调较大模型的场景。

---

### 38. QLoRA 会不会影响效果？

**答案：**

可能会有轻微影响，但在很多任务中效果接近 LoRA 或全参数微调。影响取决于模型、数据、量化方式和任务复杂度。

如果任务对数值精度非常敏感，或者需要大幅改变模型能力，全参数微调可能更稳。但大多数指令微调、格式微调和领域适配任务，QLoRA 是很实用的选择。

---

### 39. 训练后的 LoRA 权重如何部署？

**答案：**

有两种常见方式：

1. 保留 base model + LoRA adapter，推理时动态加载 adapter。
2. 将 LoRA 权重 merge 到 base model，导出合并后的模型。

动态加载适合多任务、多客户、多版本 adapter 管理。合并权重适合单一模型部署，推理更简单。

---

### 40. 多个 LoRA adapter 可以组合吗？

**答案：**

理论上可以，但要谨慎。不同 adapter 可能学习了不同任务、风格或领域，直接相加可能冲突。

可行方式包括：

- 按任务动态切换 adapter。
- 做 adapter merging。
- 使用路由器选择 adapter。
- 对合并后的模型重新评估。

不能简单认为多个 adapter 组合后能力一定叠加。

---

## 五、对齐与偏好优化

### 41. RLHF 是什么？

**答案：**

RLHF 是 Reinforcement Learning from Human Feedback，用人类反馈训练模型。典型流程是：

1. SFT 得到初始指令模型。
2. 收集人类偏好数据。
3. 训练 reward model。
4. 用强化学习优化模型，使输出获得更高 reward。

RLHF 能让模型更符合人类偏好，但工程复杂、训练不稳定、成本高。

---

### 42. DPO 是什么？

**答案：**

DPO 是 Direct Preference Optimization，直接偏好优化。它使用 chosen/rejected 成对数据，不需要单独训练 reward model，也不需要复杂 RL 过程。

数据形式：

```json
{
  "prompt": "解释 LoRA",
  "chosen": "准确、简洁、有结构的回答",
  "rejected": "错误、啰嗦或不完整的回答"
}
```

DPO 的优势是实现简单、稳定性较好，在很多偏好对齐场景中很常用。

---

### 43. SFT 和 DPO 的区别是什么？

**答案：**

SFT 学习“标准答案是什么”，DPO 学习“两个答案哪个更好”。

SFT 适合建立基本任务能力和输出格式。DPO 适合进一步优化偏好，例如更简洁、更安全、更符合人类偏好。

实际流程通常是：

```text
Base Model → SFT → DPO / RLHF → Evaluation → Deployment
```

---

### 44. 偏好数据怎么构造？

**答案：**

偏好数据通常包含同一个 prompt 下的好答案和差答案。

来源包括：

- 人工写 chosen 和 rejected。
- 多个模型生成答案，人工排序。
- 线上日志中用户采纳和拒绝行为。
- 用强模型辅助打分，再人工抽检。

关键是 chosen 和 rejected 要有明确差异，不能只是风格略有不同，否则训练信号很弱。

---

### 45. 什么是 reward hacking？

**答案：**

reward hacking 是模型学会利用 reward model 的漏洞，而不是真正变好。例如模型输出看似礼貌、很长、格式完整，但实际内容空洞。

缓解方法：

- 提高 reward 数据质量。
- 多维度评估。
- 加入人工评测。
- 避免单一 reward 指标。
- 监控模型是否出现投机输出模式。

---

### 46. 对齐训练会不会降低模型能力？

**答案：**

有可能。过度对齐可能让模型变得保守、啰嗦、拒答过多，甚至降低复杂推理和创造性。

因此对齐后必须做回归评估：

- 通用问答能力。
- 领域任务能力。
- 安全性。
- 拒答准确率。
- 有用性和简洁性。

好的对齐不是让模型少回答，而是让模型在该回答时更可靠，在不该回答时有边界。

---

## 六、评估指标与实验设计

### 47. 微调后如何评估模型？

**答案：**

评估应分层：

- 自动指标：准确率、F1、BLEU、ROUGE、Exact Match、JSON valid rate。
- LLM-as-a-judge：用强模型评估回答质量，但要防偏差。
- 人工评估：专家打分、业务人员验收。
- 线上指标：转化率、解决率、人工接管率、用户满意度。
- 安全评估：越权、隐私泄露、幻觉、拒答。

不能只看一个指标，要结合业务目标。

---

### 48. 如何评估输出格式稳定性？

**答案：**

可以设计格式专用指标：

- JSON parse success rate。
- 必填字段完整率。
- schema validation pass rate。
- 正则匹配通过率。
- 字段类型正确率。
- 多轮对话中格式保持率。

如果格式非常关键，除了微调，还可以结合 constrained decoding、函数调用、后处理校验和重试机制。

---

### 49. 如何评估幻觉？

**答案：**

幻觉评估要看任务类型。

文档问答中可以评估：

- 答案是否被上下文支持。
- 是否引用不存在的信息。
- 引用位置是否正确。
- 不知道时是否承认不知道。

领域问答中可以让专家标注事实正确性。对于 RAG 系统，还要拆分评估检索错误和生成错误。

---

### 50. 为什么要保留一个固定评测集？

**答案：**

固定评测集用于比较不同模型、不同训练数据、不同超参数的效果。如果每次评测样本都变，就无法判断改动是否真的有效。

评测集应包含：

- 高频真实问题。
- 长尾问题。
- 边界输入。
- 安全风险输入。
- 格式压力测试。
- 多轮对话样本。

---

### 51. 什么是 ablation study？

**答案：**

消融实验是指一次只去掉或改变一个因素，观察效果变化。例如：

- 去掉合成数据。
- 只训练 attention LoRA。
- 改变 LoRA rank。
- 去掉拒答样本。
- 不混入通用数据。

消融实验能回答“到底是哪部分带来了提升”。

---

### 52. 如何判断微调收益是否值得？

**答案：**

要比较收益和成本：

- 效果提升是否显著。
- 是否优于 prompt/RAG/规则方案。
- 训练和部署成本是否可接受。
- 维护数据和模型版本的成本。
- 是否带来安全或合规风险。
- 是否能在真实业务指标上提升。

如果微调只提升了离线指标，却没有提升线上业务指标，就要重新审视任务定义和评估方式。

---

## 七、部署与工程落地

### 53. 微调模型部署前要检查什么？

**答案：**

至少检查：

- 模型权重和 tokenizer 是否匹配。
- prompt template 是否和训练一致。
- LoRA 是否正确加载或合并。
- 推理精度和量化方式是否影响效果。
- 上下文长度是否满足业务需求。
- 延迟、吞吐、显存是否达标。
- 安全策略和日志监控是否完善。

---

### 54. 为什么 tokenizer 很重要？

**答案：**

tokenizer 决定文本如何切分成 token。如果训练和推理 tokenizer 不一致，模型输入分布会错位，轻则效果下降，重则输出异常。

如果扩展特殊 token，例如 `<tool_call>`、`<|assistant|>`，需要同时 resize embedding，并确保训练和推理模板一致。

---

### 55. 微调模型上线后如何监控？

**答案：**

可以监控：

- 请求量、延迟、错误率。
- 输出长度和截断率。
- JSON 解析失败率。
- 用户满意度和人工接管率。
- 幻觉投诉率。
- 拒答率。
- 安全拦截率。
- 数据分布漂移。

大模型上线不是结束，而是持续收集失败样本、更新数据、再训练和再评估。

---

### 56. 模型量化会影响微调模型吗？

**答案：**

会。量化可以降低推理成本，但可能影响精度，尤其是复杂推理、数学、代码和格式严格任务。

常见策略：

- 先用原精度评估基线。
- 再评估 INT8、INT4 等量化版本。
- 对关键任务做回归测试。
- 如果 LoRA 未合并，要确认推理框架支持量化 base + adapter。

---

### 57. 微调后的模型为什么线上效果不如离线？

**答案：**

可能原因：

- 离线测试集不代表真实线上分布。
- prompt template 线上线下不一致。
- 检索或工具链线上有差异。
- 解码参数不同，例如 temperature 太高。
- 用户输入更脏、更长、更模糊。
- 线上有安全过滤或截断。
- 评估指标和业务指标不一致。

排查时要先复现同一条线上请求，比较输入、prompt、模型版本、adapter、解码参数和后处理。

---

### 58. 如何做模型版本管理？

**答案：**

需要管理：

- base model 版本。
- tokenizer 版本。
- 训练数据版本。
- LoRA adapter 版本。
- 训练代码和超参数。
- 评估报告。
- 部署配置。

每次上线应能追溯：这个模型用什么数据、什么参数、什么代码训练出来，在哪些评测集上表现如何。

---

### 59. 多租户场景如何使用微调？

**答案：**

多租户场景可以考虑：

- 共用一个 base model。
- 每个客户一个 LoRA adapter。
- 根据租户动态加载 adapter。
- 对敏感客户隔离训练数据和模型存储。
- 通过权限控制避免 adapter 混用。

如果客户之间差异不大，也可以用 prompt/RAG 做轻量定制，避免维护过多 adapter。

---

### 60. 如何降低微调模型推理成本？

**答案：**

常见方法：

- 使用更小的 base model。
- 量化推理。
- 合并 LoRA 权重。
- KV cache。
- batch 推理。
- speculative decoding。
- 缩短 prompt。
- 用分类器或路由器决定是否调用大模型。
- 对简单任务使用小模型或规则。

---

## 八、常见故障排查

### 61. 微调后模型总是输出很长怎么办？

**答案：**

可能原因：

- 训练数据答案普遍过长。
- 没有明确长度约束。
- 解码参数设置不当。
- 缺少简洁回答样本。

解决方法：

- 加入短答案样本。
- 在 system prompt 和数据中明确长度要求。
- 调整 max tokens、temperature、top_p。
- 用偏好数据让模型学习“简洁答案优于啰嗦答案”。

---

### 62. 微调后模型格式经常错怎么办？

**答案：**

先排查：

- 训练数据格式是否一致。
- 是否混入错误格式答案。
- prompt template 是否与训练一致。
- 是否对 assistant 部分正确计算 loss。
- 推理温度是否太高。

如果格式非常关键，建议结合：

- JSON schema 校验。
- constrained decoding。
- 函数调用。
- 失败重试。
- 格式错误样本补充训练。

---

### 63. 训练 loss 很低但效果不好，为什么？

**答案：**

可能原因：

- 训练集太简单或重复。
- 测试集分布不同。
- 模型过拟合。
- loss 只反映 token 预测，不反映业务质量。
- 数据里有模板答案，模型学会了模板但没学会任务。

所以评估不能只看 loss，要看任务指标和真实样本表现。

---

### 64. 微调后模型变笨了怎么办？

**答案：**

这可能是灾难性遗忘或过拟合。

可尝试：

- 降低学习率。
- 减少 epoch。
- 增加通用指令混合数据。
- 使用 LoRA 而不是全参数微调。
- 减小 LoRA rank 或 target modules。
- 检查训练数据是否过窄、重复或低质量。

---

### 65. 微调后模型拒答过多怎么办？

**答案：**

原因可能是安全拒答数据比例过高，或拒答边界不清晰。

解决方法：

- 区分应答、澄清、拒答三类样本。
- 减少泛化过强的拒答模板。
- 增加“看似敏感但可回答”的样本。
- 用偏好数据优化拒答边界。
- 建立拒答准确率评估集。

---

### 66. 为什么模型学不会工具调用？

**答案：**

可能原因：

- 工具调用格式不稳定。
- 缺少 tool result 后的 final answer 样本。
- 训练数据没有覆盖失败、重试、参数缺失场景。
- 推理时工具 schema 和训练时不一致。
- 模型没有被训练何时调用工具、何时直接回答。

工具调用数据要覆盖完整链路：

```text
用户请求 → 判断需要工具 → 生成工具参数 → 接收工具结果 → 总结给用户
```

---

### 67. 为什么模型多轮对话能力差？

**答案：**

可能原因：

- 训练数据主要是单轮。
- 多轮样本中上下文关系弱。
- 历史消息截断策略不合理。
- 训练和推理的 chat template 不一致。
- 没有覆盖追问、纠错、改口、补充信息场景。

要提升多轮能力，需要构造真实多轮任务轨迹，而不是把单轮样本简单拼接。

---

### 68. 微调后模型出现重复输出怎么办？

**答案：**

可能原因：

- 数据中存在重复段落。
- 学习率过高。
- 训练过拟合。
- 解码参数不合理。
- EOS token 处理错误。

排查重点：

- 检查 tokenizer 和 EOS 设置。
- 检查训练样本是否有重复答案。
- 降低 temperature 或设置 repetition penalty。
- 减少训练轮数。

---

## 九、场景题

### 69. 如果让你微调一个法律合同审查模型，你会怎么做？

**答案：**

我会先明确任务边界，例如风险条款识别、主体信息抽取、审查意见生成，而不是泛泛“懂法律”。

流程：

1. 收集真实合同和律师审查意见，做脱敏。
2. 定义输出 schema，例如风险类型、条款位置、风险等级、修改建议。
3. 构造 SFT 数据，让模型学会结构化输出。
4. 对高风险问题构造拒答和免责声明。
5. 评估字段准确率、风险召回率、误报率和人工可用性。
6. 结合 RAG 检索法律法规和公司模板。
7. 上线后收集律师修订记录，持续迭代。

---

### 70. 如果客服模型回答风格不符合品牌要求，你会微调吗？

**答案：**

先不一定微调。可以先尝试 system prompt、few-shot 示例和后处理模板。如果风格要求稳定、场景多、prompt 难以控制，再考虑 SFT。

微调方案：

- 收集优秀客服对话。
- 标注不同场景的标准回复。
- 加入反例，例如过度承诺、语气生硬、违规表达。
- 微调后评估解决率、满意度、合规率和人工接管率。

---

### 71. 如果模型在企业知识问答中经常编造答案，应该微调还是 RAG？

**答案：**

优先 RAG。企业知识通常更新频繁，微调不适合记忆动态知识。微调可以用于让模型学会回答规范，例如引用来源、不知道时拒答、按固定格式输出。

较好的方案是：

```text
RAG 提供知识
SFT 提供回答行为
评估集约束引用和事实一致性
```

---

### 72. 如果只有 500 条高质量数据，能不能微调？

**答案：**

可以试，但要谨慎。

适合场景：

- 输出格式学习。
- 风格适配。
- 简单分类或抽取。
- LoRA 小规模训练。

不适合期待模型学会复杂新能力。可以结合数据增强、few-shot、RAG 和人工评估。训练时要降低过拟合风险，保留验证集，少量 epoch 试跑。

---

### 73. 如何设计一次微调实验？

**答案：**

我会这样设计：

1. 定义目标指标和失败样本。
2. 建立 baseline，例如原模型 + prompt。
3. 准备训练集、验证集、测试集。
4. 先小规模 LoRA 试跑。
5. 比较不同学习率、rank、epoch。
6. 做人工评估和安全评估。
7. 做消融实验验证数据和参数贡献。
8. 输出实验报告，记录数据版本、代码版本和模型版本。

---

### 74. 如果线上数据持续变化，微调模型如何迭代？

**答案：**

需要建立闭环：

- 收集线上失败样本。
- 分类失败原因：知识缺失、格式错误、推理错误、安全问题。
- 对知识缺失优先更新 RAG。
- 对行为模式问题加入微调数据。
- 定期训练候选模型。
- 固定评测集 + 新增回归集评估。
- 灰度上线，监控核心指标。

---

### 75. 面试中如何一句话解释微调的核心？

**答案：**

可以这样说：

> 微调的核心不是把知识硬塞进模型，而是用高质量样本把模型的行为分布从“通用回答”调整到“符合特定任务、格式、领域和偏好”的分布。

这句话能体现对微调目标、数据质量和模型行为的理解。

---

## 十、简短速记

- 补知识：优先 RAG。
- 学格式：SFT 很有效。
- 学偏好：DPO/RLHF。
- 降成本：LoRA/QLoRA。
- 防遗忘：小学习率、少 epoch、混通用数据、做回归评估。
- 看效果：不要只看 loss，要看业务指标。
- 要上线：版本、评估、监控、回滚一个都不能少。
