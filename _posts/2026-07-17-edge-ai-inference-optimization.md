> 一句话总结：端侧推理加速不是单纯把模型量化一下，而是在质量、延迟、内存、功耗和硬件兼容之间做系统工程取舍。

![端侧 AI 推理链路](/assets/images/edge-ai-inference/edge-inference-pipeline.svg)

<!--more-->

AI 正在从云端走向端侧。

手机、车机、机器人、智能摄像头、AR 眼镜、可穿戴设备、工业网关，都开始运行越来越复杂的模型。端侧 AI 的好处很明显：低延迟、保护隐私、弱网可用、节省云端成本，也能让产品体验更自然。

但端侧推理也很难。

云端可以堆 GPU，端侧设备却只有有限的 CPU、GPU、NPU、DSP、内存和电池。一个在服务器上毫无压力的模型，放到手机或嵌入式设备上，可能直接遇到五个问题：

1. 模型太大，放不进内存。
2. 推理太慢，用户等不起。
3. 功耗太高，设备发热降频。
4. 算子不支持，NPU 跑不了，只能退回 CPU。
5. 精度下降，优化后模型变快但效果崩了。

所以端侧推理优化，本质上是把一个“能跑的模型”变成一个“在真实设备上可用的模型”。

## 一、端侧推理优化到底优化什么？

端侧优化通常围绕四个目标：

| 目标 | 解释 | 常见指标 |
| --- | --- | --- |
| 更快 | 用户感知延迟更低，帧率更稳 | 首帧延迟、单次推理耗时、FPS、P95 |
| 更小 | 模型和运行时占用更少内存 | 模型体积、峰值内存、显存/NPU 内存 |
| 更省电 | 降低发热和电池消耗 | 功耗、温度、降频次数 |
| 更稳 | 不同设备、不同输入下表现稳定 | 崩溃率、超时率、质量回归 |

这里最容易误解的是：**速度不是唯一目标。**

如果为了加速把模型量化到 INT4，结果识别准确率下降 10%，或者回答质量明显变差，那这个优化在产品上可能是失败的。端侧优化必须先定义质量底线，再在质量底线内追求速度和成本。

## 二、端侧优化的三角约束：延迟、内存、功耗

![端侧优化的三角约束](/assets/images/edge-ai-inference/latency-memory-power.svg)

端侧设备和云端服务器最大的差异，是资源约束更强，而且这些资源互相牵制。

| 约束 | 典型问题 |
| --- | --- |
| 延迟 | 用户点击后多久看到结果？语音助手多久开始回复？摄像头能否实时处理？ |
| 内存 | 模型权重、激活、KV Cache、输入缓存能否放得下？ |
| 功耗 | 设备是否发热？是否触发降频？电池能撑多久？ |

不同产品的优先级不同：

| 场景 | 优先目标 | 优化重点 |
| --- | --- | --- |
| 语音助手 | 首响应低延迟 | 流式输出、小模型路由、缓存 |
| 手机相册离线分析 | 低功耗、批处理 | 后台调度、低优先级运行、NPU |
| 智能摄像头 | 稳定 FPS | 模型轻量化、算子融合、流水线 |
| 车载感知 | 确定性延迟 | 实时调度、硬件绑定、冗余策略 |
| 端侧 LLM | 内存与生成速度 | 量化、KV Cache、推测解码 |
| AR 眼镜 | 帧率和功耗 | 小模型、低分辨率、异步推理 |

所以端侧优化不能脱离场景。没有“所有场景最优”的方案，只有“在这个产品约束下最合适”的方案。

## 三、第一层优化：模型本身变小、变快

模型层优化是最直观的一层。

### 1. 量化：最常用的端侧优化手段

量化把模型权重和计算从 FP32/FP16 降到 INT8、INT4、FP8 等更低精度。

| 量化类型 | 说明 | 优点 | 风险 |
| --- | --- | --- | --- |
| FP16/BF16 | 半精度浮点 | 简单、质量稳定 | 压缩有限 |
| INT8 | 8 bit 整数量化 | 端侧支持广，速度和体积收益明显 | 需要校准，部分模型退化 |
| INT4 | 4 bit 权重量化 | 体积更小，适合端侧 LLM | 质量更敏感，算子支持不一 |
| 混合精度 | 不同层用不同精度 | 在质量和速度间平衡 | 工程复杂 |

量化又分为：

| 方法 | 说明 |
| --- | --- |
| PTQ | Post Training Quantization，训练后量化，不重新训练或少量校准 |
| QAT | Quantization Aware Training，训练时模拟量化误差，质量更稳 |
| Weight-only Quantization | 只量化权重，常用于 LLM |
| Activation Quantization | 同时量化激活，速度收益更大但更难 |

端侧项目里，通常先试 PTQ，因为成本最低；如果质量退化明显，再考虑 QAT 或混合精度。

### 2. 蒸馏：让小模型学大模型

蒸馏的思路是用大模型或强模型生成软标签、解释过程或中间表示，再训练一个更小的学生模型。

端侧很适合蒸馏，因为很多设备上根本跑不了大模型。与其硬塞一个大模型，不如训练一个针对具体任务的小模型。

| 蒸馏对象 | 适用场景 |
| --- | --- |
| 分类 logits | 图像分类、意图识别 |
| 中间特征 | 检测、分割、多模态编码 |
| 推理过程 | 小型语言模型、数学/代码任务 |
| 偏好输出 | 端侧助手回答风格 |

蒸馏的核心不是“学生模型越小越好”，而是让学生模型只学产品真正需要的能力。

### 3. 剪枝和稀疏：减少无效计算

剪枝会移除不重要的通道、层、注意力头或权重。

| 方法 | 说明 |
| --- | --- |
| 非结构化剪枝 | 去掉单个权重，稀疏度高但硬件不一定加速 |
| 结构化剪枝 | 去掉通道、层、头，更容易实际加速 |
| N:M 稀疏 | 每 M 个权重保留 N 个，适配特定硬件 |

剪枝的难点是：理论 FLOPs 下降，不代表真实延迟一定下降。端侧硬件更喜欢规则、连续、易并行的结构。结构化剪枝通常比非结构化剪枝更容易带来实际收益。

### 4. 架构替换：从源头选择端侧友好模型

有时候，优化一个不适合端侧的模型，不如直接换架构。

| 任务 | 端侧友好方向 |
| --- | --- |
| 图像分类 | MobileNet、EfficientNet-Lite、ConvNeXt-Tiny |
| 目标检测 | YOLO-Nano、RT-DETR 小模型、Mobile-friendly detector |
| 语音识别 | 小型 Conformer、流式 ASR |
| 端侧 LLM | 小参数模型、GQA/MQA、短上下文、量化友好结构 |
| 多模态 | 轻量视觉 encoder + 小语言模型 |

端侧模型设计的第一原则是：**不要让后端优化为不合理架构买单。**

## 四、第二层优化：算子融合与图优化

模型结构确定后，Runtime 会把模型图转换成设备可执行的计算图。

图优化常见手段包括：

| 优化 | 作用 |
| --- | --- |
| 算子融合 | Conv + BN + ReLU 合并，减少内存读写 |
| 常量折叠 | 提前计算不变的子图 |
| 死节点删除 | 移除无用计算 |
| Layout 转换 | 选择硬件更高效的数据布局，如 NHWC/NCHW |
| Kernel 选择 | 为不同 shape 选择最快实现 |
| 内存复用 | 复用中间 tensor buffer，降低峰值内存 |

很多时候，推理慢不是算力不够，而是内存搬运太多。

例如 Conv、BN、Activation 分开执行时，每一步都要读写中间结果；融合后可以在一个 kernel 里完成，减少访存和调度开销。

端侧 runtime 的差异也很大：

| Runtime | 常见场景 |
| --- | --- |
| TensorFlow Lite | Android、移动端、嵌入式 |
| Core ML | Apple 生态 |
| ONNX Runtime | 跨平台部署 |
| MNN | 移动端、阿里生态 |
| NCNN | 移动端、轻量视觉模型 |
| TVM | 编译优化和硬件适配 |
| TensorRT / TensorRT-LLM | NVIDIA GPU 或边缘设备 |
| OpenVINO | Intel CPU/GPU/NPU |

选择 Runtime 时不能只看 benchmark，还要看：

1. 目标设备是否支持。
2. 模型算子是否完整支持。
3. NPU/GPU delegate 是否稳定。
4. 量化格式是否匹配。
5. 工具链是否方便调试。

## 五、第三层优化：内存管理与 KV Cache

端侧 LLM 的最大瓶颈之一是内存。

LLM 推理通常分为两个阶段：

| 阶段 | 特点 |
| --- | --- |
| Prefill | 一次性处理输入 prompt，计算量大，可并行 |
| Decode | 逐 token 生成，每步依赖前一步，受内存访问和 KV Cache 影响 |

KV Cache 会保存历史 token 的 Key/Value，避免每生成一个 token 都重新计算整段上下文。

但 KV Cache 也会吃掉大量显存/内存：

```text
KV Cache 大小 ≈ 层数 × token 数 × hidden 维度 × 精度 × 2(K/V)
```

端侧设备内存有限，因此需要优化：

| 方法 | 作用 |
| --- | --- |
| 限制上下文长度 | 直接控制 KV Cache 增长 |
| Sliding Window Attention | 只保留最近窗口 |
| KV Cache 量化 | 用 INT8/INT4 存 KV |
| GQA/MQA | 多个 query head 共享更少 KV head |
| Prefix Cache | 复用系统 prompt 或固定前缀 |
| Context Compression | 压缩历史对话 |

端侧 LLM 一定要关注上下文策略。无限长上下文在端侧通常不现实，产品上要设计摘要、裁剪、分层记忆或云端协同。

## 六、第四层优化：推测解码和小模型路由

LLM 的 decode 阶段是逐 token 的，很难完全并行。推测解码可以缓解这个问题。

它的思路是：

1. 用一个小模型快速生成多个候选 token。
2. 用大模型一次性验证这些 token。
3. 如果验证通过，就一次接受多个 token。
4. 如果不通过，就回退到大模型输出。

这样可以减少大模型调用次数，提高生成速度。

端侧推测解码适合：

| 场景 | 原因 |
| --- | --- |
| 小模型和大模型都在端侧 | 避免网络延迟 |
| 小模型非常快 | 草稿生成成本低 |
| 输出相对可预测 | 验证通过率高 |
| 对质量要求高 | 最终仍由大模型校验 |

但它也有成本：

1. 需要同时加载两个模型。
2. 小模型太弱会导致通过率低。
3. 工程实现复杂。
4. 对内存更敏感。

另一种常见策略是模型路由：

| 请求类型 | 路由策略 |
| --- | --- |
| 简单分类、固定指令 | 小模型 |
| 高风险或复杂推理 | 大模型或云端 |
| 离线批处理 | 低功耗后台模型 |
| 实时交互 | 低延迟模型 |

端侧产品不一定所有请求都跑同一个模型。合理路由往往比单模型极致优化更有效。

## 七、第五层优化：系统调度与硬件协同

端侧设备通常有异构硬件：

| 硬件 | 优点 | 局限 |
| --- | --- | --- |
| CPU | 通用、兼容好、适合控制逻辑 | 矩阵计算慢、功耗高 |
| GPU | 并行能力强 | 调度开销、功耗、内存竞争 |
| NPU | 高能效，适合神经网络 | 算子支持有限，工具链碎片化 |
| DSP | 适合音频、信号处理 | 编程和调试复杂 |
| ISP | 图像前处理高效 | 任务受限 |

端侧优化要让不同硬件各做擅长的事。

比如智能摄像头：

```text
ISP 做图像预处理
→ NPU 跑检测模型
→ CPU 做后处理和业务逻辑
→ 必要时 GPU 做渲染
```

系统层还要关注：

| 优化点 | 说明 |
| --- | --- |
| 线程绑定 | 避免关键任务被调度到低性能核心 |
| 大小核策略 | 大核跑高优先级任务，小核跑后台任务 |
| 预处理流水线 | 输入处理和模型推理并行 |
| 零拷贝 | 减少 CPU/GPU/NPU 之间数据搬运 |
| Warmup | 避免首次推理冷启动过慢 |
| Thermal Control | 控制温度，避免长时间降频 |
| Background Scheduling | 离线任务放到充电、空闲或低温时执行 |

端侧很多性能问题不是模型问题，而是系统调度问题。

## 八、端侧 LLM 的特殊优化点

端侧 LLM 和端侧视觉模型不太一样。

视觉模型常常是固定输入、固定输出，优化目标是 FPS 和单帧延迟。LLM 则是变长输入、变长输出、多轮上下文、逐 token 生成。

端侧 LLM 需要特别关注：

| 问题 | 优化方向 |
| --- | --- |
| 模型太大 | 选择 0.5B-3B 小模型、INT4 权重量化 |
| 首 token 慢 | Prompt 缓存、短 prompt、预填充优化 |
| 生成慢 | 推测解码、GQA/MQA、Kernel 优化 |
| 上下文太长 | 摘要、滑窗、分层记忆 |
| 内存爆炸 | KV Cache 量化、限制并发 |
| 输出质量下降 | 蒸馏、DPO、端侧任务专训 |
| 发热降频 | 限制最大 token、动态降级 |

一个实用原则是：端侧 LLM 不要试图成为万能大模型，而要成为场景模型。

例如：

| 场景 | 更合理的端侧定位 |
| --- | --- |
| 手机助手 | 短指令理解、隐私敏感内容本地处理 |
| 输入法 | 改写、补全、风格转换 |
| 车机 | 简短语音指令和本地控制 |
| 可穿戴设备 | 健康摘要、提醒、轻量问答 |
| 机器人 | 本地任务规划和状态解释 |

复杂开放问答仍然可以走云端或云边协同。

## 九、端侧优化的评测方法

端侧优化不能只在开发机上测。

必须在真实目标设备上测，并且至少覆盖：

| 维度 | 测什么 |
| --- | --- |
| 冷启动 | 第一次加载模型和首次推理耗时 |
| 热启动 | 模型已加载后的推理耗时 |
| P50/P95/P99 | 不只看平均值，看尾部延迟 |
| 峰值内存 | 权重、激活、缓存、输入输出全部算上 |
| 长时间运行 | 是否发热、降频、内存泄漏 |
| 不同输入 | 短文本/长文本、小图/大图、简单/复杂场景 |
| 质量回归 | 量化或裁剪后准确率、召回率、回答质量变化 |
| 多任务干扰 | 与摄像头、渲染、音频、系统服务同时运行 |

建议建立一个端侧 benchmark 表：

| 版本 | 模型大小 | 精度 | 设备 | 延迟 P50 | 延迟 P95 | 峰值内存 | 功耗 | 质量 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| baseline | FP16 | 未优化 | A 设备 | 120ms | 210ms | 900MB | 高 | 100% |
| int8 | INT8 | PTQ | A 设备 | 72ms | 130ms | 520MB | 中 | 98.5% |
| int4 | INT4 | PTQ | A 设备 | 55ms | 110ms | 330MB | 低 | 94.0% |
| distill | INT8 | 蒸馏小模型 | A 设备 | 38ms | 80ms | 260MB | 低 | 97.2% |

这种表比单独一句“快了 2 倍”更有价值，因为它能体现质量和资源之间的 trade-off。

## 十、端侧优化的常见坑

端侧推理优化里，有几个坑非常常见。

| 坑 | 说明 | 建议 |
| --- | --- | --- |
| 只看 FLOPs | FLOPs 下降不代表真实设备变快 | 以真机延迟为准 |
| 只测平均延迟 | 平均值好看但 P99 卡顿 | 看 P50/P95/P99 |
| 忽略预处理 | resize、tokenize、decode 可能占很大比例 | 端到端 profiling |
| 量化不做校准 | 直接量化导致质量大幅下降 | 建校准集和回归集 |
| NPU 算子不支持 | 部分算子回退 CPU，反而更慢 | 检查 delegate partition |
| 忽略热降频 | 短测很快，长跑后变慢 | 做长时间压力测试 |
| 没有质量门槛 | 速度提升但效果不可用 | 先定质量下限 |
| 缺少设备分层 | 高端机可用，低端机崩溃 | 按设备能力分策略 |

端侧优化最重要的工具是 profiling。不要猜瓶颈在哪里，要用数据看：

1. 每个算子耗时。
2. CPU/GPU/NPU 占用。
3. 内存峰值。
4. 数据拷贝耗时。
5. 冷启动耗时。
6. 长时间运行温度和频率。

## 十一、一个实用优化流程

如果要优化一个端侧模型，我建议按这个顺序：

| 步骤 | 目标 |
| --- | --- |
| 1. 定义产品约束 | 设备、延迟、内存、功耗、质量下限 |
| 2. 建立 baseline | 在真机上测端到端性能 |
| 3. 做 profiling | 找到真正瓶颈：模型、预处理、算子、内存还是调度 |
| 4. 先做低风险优化 | 图优化、算子融合、线程、缓存、预处理流水线 |
| 5. 再做量化 | PTQ → QAT → 混合精度 |
| 6. 再考虑模型替换 | 蒸馏、剪枝、小模型架构 |
| 7. 做长时间稳定性测试 | 发热、降频、内存泄漏 |
| 8. 做质量回归 | 对比优化前后效果 |
| 9. 分设备发布 | 高端机、低端机策略不同 |
| 10. 上线监控 | 收集延迟、崩溃、质量反馈 |

这套流程的核心是：先测，再优化；先系统，再模型；先质量门槛，再追求速度。

## 十二、总结：端侧推理加速是系统工程

端侧 AI 推理优化不是一个单点技术，而是一组组合拳：

1. 模型层：量化、蒸馏、剪枝、轻量架构。
2. Runtime 层：图优化、算子融合、内存复用、delegate。
3. 内存层：KV Cache、激活复用、上下文裁剪。
4. 系统层：线程调度、异构硬件、零拷贝、热控制。
5. 产品层：云边协同、模型路由、缓存、降级策略。
6. 评测层：真机 benchmark、质量回归、长时间稳定性测试。

真正好的端侧优化，不是把某个 benchmark 跑到极限，而是在真实设备、真实输入、真实用户场景下，让模型稳定、快速、低功耗地工作。

端侧 AI 的未来会越来越重要。因为很多智能体验，只有发生在本地，才足够快、足够私密、足够自然。而能把模型放进端侧，并让它跑得好，就是 AI 工程能力的一块硬骨头。
