> 一句话总结：AI infra 不是“买几张 GPU 跑模型”，而是一整套把数据、训练、评测、部署、推理、监控和反馈串起来的工程体系。

![AI Infra 基础能力地图](/assets/images/ai-infra-basics/ai-infra-map.svg)

<!--more-->

过去几年，大模型让很多人重新认识了 AI。

以前我们讨论 AI，常常是在讨论模型结构：Transformer、CNN、RNN、Diffusion、MoE。后来大家开始讨论训练方法：预训练、SFT、RLHF、DPO、蒸馏、量化。再往后，真正决定一个 AI 产品能不能跑起来、能不能稳定服务、能不能持续迭代的东西浮出水面：**AI infra**。

AI infra 可以理解成 AI 系统的工程底座。

它关心的问题不是“模型能不能在 notebook 里跑通”，而是：

1. 数据能不能持续进入训练和评测流程？
2. 模型训练能不能稳定利用 GPU 集群？
3. 实验结果能不能复现、比较和回滚？
4. 模型能不能低延迟、高并发、低成本地服务用户？
5. 线上效果变差时，能不能定位是数据、模型、prompt、检索还是服务问题？
6. 用户反馈能不能回流成下一轮训练数据？

这篇文章就用尽量基础但完整的方式，梳理 AI infra 的核心概念。

## 一、AI Infra 到底是什么？

AI infra 是支撑 AI 模型全生命周期的基础设施和工程平台。

如果用传统软件来类比，Web infra 让一个网站从代码变成线上服务：代码仓库、CI/CD、数据库、缓存、负载均衡、监控、日志、灰度发布。AI infra 则让一个模型从实验变成智能产品：数据管道、训练集群、模型仓库、评测平台、推理服务、向量数据库、在线监控、反馈闭环。

可以粗略分成七层：

| 层级 | 核心问题 | 典型组件 |
| --- | --- | --- |
| 计算层 | GPU/CPU/网络/存储怎么调度？ | GPU 集群、Kubernetes、Slurm、对象存储、高速网络 |
| 数据层 | 数据怎么采集、清洗、标注、版本化？ | Data Lake、ETL、Labeling、Feature Store、Vector DB |
| 训练层 | 模型怎么高效训练和微调？ | PyTorch、DeepSpeed、FSDP、Ray、训练调度器 |
| 模型层 | 模型产物怎么管理？ | Model Registry、Checkpoint、Adapter、Prompt 版本 |
| 评测层 | 怎么判断模型变好了还是变坏了？ | Benchmark、Eval、红队、安全评测、A/B Test |
| 推理层 | 模型怎么在线服务？ | vLLM、Triton、TensorRT-LLM、SGLang、Serving Router |
| 观测与反馈层 | 线上问题怎么发现和回流？ | Logs、Tracing、Metrics、Feedback、数据回流 |

AI infra 的价值，是把这些环节从“手工脚本”变成“可重复、可观测、可规模化”的系统。

## 二、为什么大模型时代更需要 AI Infra？

小模型时代，很多团队也做 MLOps，但压力相对可控。大模型时代，infra 的重要性被放大了。

原因主要有四个。

### 1. 训练成本极高

大模型训练不是单机跑几个小时，而是成百上千张 GPU 跑数天、数周甚至更久。任何一个节点失败、网络抖动、数据加载瓶颈，都可能造成巨额资源浪费。

训练 infra 必须解决：

| 问题 | 具体表现 |
| --- | --- |
| GPU 利用率 | GPU 等数据、等通信、等 checkpoint |
| 分布式通信 | 多机多卡之间梯度同步慢 |
| 容错恢复 | 训练中断后能不能从 checkpoint 恢复 |
| 数据吞吐 | 存储读不动，导致 GPU 空转 |
| 实验管理 | 同一批实验参数、数据、代码能否复现 |

所以训练 infra 不是“能跑就行”，而是要把昂贵 GPU 资源尽量吃满。

### 2. 推理成本持续发生

训练是一次性大开销，推理是每天都在发生的开销。

一个模型上线后，每个请求都会消耗算力、显存、网络和存储。用户越多，成本越敏感。尤其是 LLM 推理，长上下文、多轮对话、并发请求都会显著影响成本。

推理 infra 需要关注：

| 指标 | 含义 |
| --- | --- |
| TTFT | Time To First Token，首 token 延迟 |
| TPOT | Time Per Output Token，每个输出 token 的平均延迟 |
| QPS | 每秒请求数 |
| P95/P99 | 高分位延迟，代表尾部体验 |
| GPU Memory | 显存占用，特别是 KV Cache |
| Cost per Request | 单次请求成本 |

一个模型实验室里效果很好，但如果线上推理成本太高、延迟太大，也很难变成产品。

### 3. AI 系统有更强的不确定性

传统软件更像确定性系统：输入 A，代码逻辑固定，输出 B。AI 系统则不完全一样。

同一个模型在不同数据、prompt、检索结果、采样参数下，输出可能不同。线上问题可能来自很多地方：

| 问题来源 | 示例 |
| --- | --- |
| 数据 | 训练数据过期、标签噪声、线上分布漂移 |
| 模型 | 新版本退化、幻觉增加、推理能力下降 |
| Prompt | 模板变更导致输出格式不稳定 |
| 检索 | RAG 召回错误、知识库版本不一致 |
| Serving | 批处理策略改变、量化精度损失 |
| 用户 | 提问方式变化、恶意 prompt、长上下文污染 |

这意味着 AI infra 必须非常重视评测、监控和回放。

### 4. AI 产品需要持续学习

AI 产品不是上线一次就结束。用户会不断提出新问题，业务规则会变，知识库会更新，安全要求也会变化。

![AI 系统的工程闭环](/assets/images/ai-infra-basics/ai-system-loop.svg)

成熟的 AI infra 应该能把线上反馈变成下一轮改进材料：

```text
用户请求 → 模型回答 → 用户反馈 / 人工审核 → 数据回流 → 评测集更新 → 模型或 prompt 迭代 → 灰度上线
```

这就是 AI 系统的工程闭环。

## 三、训练 Infra：让模型高效学会能力

![训练 Infra 与推理 Infra 的差异](/assets/images/ai-infra-basics/train-vs-infer.svg)

训练 infra 主要服务模型训练、微调和对齐。

典型训练任务包括：

| 任务 | 说明 |
| --- | --- |
| 预训练 | 用海量无标注文本、图像、视频等数据学习基础能力 |
| 继续预训练 | 在领域语料上继续训练，如医疗、法律、金融、代码 |
| SFT | 用指令数据训练模型按照用户要求回答 |
| RLHF / DPO | 用人类偏好或奖励信号优化模型行为 |
| LoRA / Adapter | 用更小成本做参数高效微调 |
| 蒸馏 | 用大模型生成数据或软标签训练小模型 |

训练 infra 的核心能力通常包括：

| 能力 | 作用 |
| --- | --- |
| 任务调度 | 把训练任务分配到合适的 GPU 资源 |
| 分布式训练 | 支持多机多卡并行 |
| 数据加载 | 高吞吐读取训练数据，减少 GPU 等待 |
| Checkpoint | 定期保存模型状态，支持失败恢复 |
| 实验追踪 | 记录参数、代码版本、数据版本、指标 |
| 资源隔离 | 防止不同团队任务互相影响 |
| 成本统计 | 统计每个实验消耗多少 GPU 小时 |

训练 infra 最怕的是“不可复现”。如果某个模型效果很好，但不知道用了哪版数据、哪版代码、哪些超参数、哪个 checkpoint，那它就很难进入工程化流程。

## 四、数据 Infra：AI 系统的燃料管道

模型能力很大程度上由数据决定。

数据 infra 要解决的不只是“把数据存起来”，还包括：

1. 数据从哪里来。
2. 数据是否合法合规。
3. 数据质量如何。
4. 数据如何清洗去重。
5. 数据如何标注。
6. 数据版本如何管理。
7. 哪个模型用了哪批数据。

常见数据类型包括：

| 数据类型 | 典型用途 |
| --- | --- |
| 预训练语料 | 建立基础语言或多模态能力 |
| 指令数据 | 让模型学会按任务回答 |
| 偏好数据 | 训练 reward model、DPO、RLHF |
| 评测数据 | 判断模型能力变化 |
| 业务知识 | 构建 RAG 知识库 |
| 用户日志 | 发现真实场景问题 |
| 人工纠错 | 形成高价值迭代数据 |

大模型时代还有一个重要组件：**向量数据库**。

向量数据库主要用于 RAG。它把文档切成 chunks，编码成向量，用户提问时检索相关内容，再交给大模型回答。

RAG 相关 infra 需要关注：

| 环节 | 常见问题 |
| --- | --- |
| 文档解析 | PDF、表格、网页结构丢失 |
| Chunk 切分 | 太短缺上下文，太长召回不准 |
| Embedding | 向量模型不适合领域语义 |
| 检索 | 召回不全、召回错误、排序差 |
| 引用 | 模型回答没有准确引用来源 |
| 更新 | 知识库变更后索引未同步 |

很多 AI 应用不是模型本身不行，而是数据和检索链路不稳定。

## 五、模型 Infra：管理 checkpoint、版本和发布

模型不是一个单独文件，而是一组产物：

| 产物 | 说明 |
| --- | --- |
| Base Model | 基座模型 |
| Checkpoint | 某次训练保存的模型权重 |
| Adapter | LoRA、Prefix、Prompt Tuning 等增量权重 |
| Tokenizer | 分词器，必须和模型匹配 |
| Config | 模型结构、上下文长度、特殊 token 等 |
| Prompt Template | 系统提示词、工具调用格式、输出规范 |
| Quantized Model | INT8、INT4、FP8 等量化版本 |

模型 infra 要解决：

1. 每个模型版本从哪里来。
2. 对应哪批训练数据和代码。
3. 评测结果如何。
4. 是否允许上线。
5. 上线后能否回滚。
6. 权限和合规如何管理。

这就需要 Model Registry。

一个好的 Model Registry 不只是文件目录，而是模型版本的元数据中心：

| 元数据 | 用途 |
| --- | --- |
| 模型名称和版本 | 区分不同模型 |
| 训练任务 ID | 追溯来源 |
| 数据版本 | 判断数据依赖 |
| 评测报告 | 上线门禁 |
| 模型卡 | 说明能力边界和风险 |
| 部署配置 | 推理框架、量化方式、资源需求 |
| 状态 | dev、staging、production、archived |

没有模型版本管理，AI 系统会很快变成一堆“final_v3_really_final.bin”。

## 六、评测 Infra：判断模型到底有没有变好

AI 系统最容易踩的坑，是“感觉变好了”。

模型更新、prompt 更新、RAG 更新，都可能让一部分问题变好，另一部分问题变差。评测 infra 的任务，就是把这种变化量化出来。

评测通常分几类：

| 评测类型 | 目的 |
| --- | --- |
| Benchmark | 测基础能力，如推理、知识、代码、数学 |
| 领域评测 | 测行业问题，如医疗、法律、电力、金融 |
| 任务评测 | 测具体产品任务，如客服、搜索、摘要 |
| 安全评测 | 测越狱、危险建议、隐私泄露、偏见 |
| 回归评测 | 确保新版不破坏旧能力 |
| 人工评测 | 对开放式问题做专家判断 |
| 在线 A/B | 在真实用户流量中比较效果 |

评测 infra 需要注意两件事。

第一，评测集要版本化。否则模型 A 和模型 B 用的评测题不一样，结果不可比。

第二，评测要接近真实场景。通用 benchmark 很有用，但产品是否好用，还要看真实用户问题。

## 七、推理 Infra：把模型变成稳定服务

推理 infra 是 AI 产品离用户最近的一层。

它负责把模型部署成 API、WebSocket、批处理任务或边缘服务。对 LLM 来说，推理 infra 远比传统模型复杂，因为生成是逐 token 的，显存里还要维护 KV Cache。

推理优化常见技术包括：

| 技术 | 作用 |
| --- | --- |
| Dynamic Batching | 把多个请求合并，提高吞吐 |
| Continuous Batching | 生成过程中动态加入新请求 |
| KV Cache | 缓存历史 token 的 key/value，减少重复计算 |
| Prefix Cache | 多请求共享相同前缀时复用缓存 |
| Quantization | 降低显存和计算成本 |
| Speculative Decoding | 用小模型草稿加速大模型生成 |
| Tensor Parallel | 多 GPU 切分单个大模型 |
| Router | 在多个模型、多个实例之间调度请求 |

推理 infra 需要同时处理两类矛盾：

| 矛盾 | 说明 |
| --- | --- |
| 延迟 vs 吞吐 | batch 越大吞吐越高，但单个请求可能等更久 |
| 成本 vs 质量 | 小模型便宜但能力弱，大模型贵但效果好 |
| 上下文长度 vs 显存 | 长上下文消耗大量 KV Cache |
| 稳定性 vs 迭代速度 | 快速上线新模型可能带来质量波动 |

所以推理服务不是简单 `model.generate()`，而是一套资源调度系统。

## 八、观测 Infra：线上 AI 系统怎么排查问题

AI 系统上线后，必须有观测能力。

传统服务监控 CPU、内存、QPS、错误率。AI 服务还要监控模型行为。

| 指标 | 说明 |
| --- | --- |
| 请求量 | 用户使用情况 |
| TTFT / TPOT | 首 token 和生成速度 |
| 输入/输出 token | 成本和上下文长度 |
| 错误率 | 服务异常、超时、工具失败 |
| 检索命中 | RAG 是否召回有效内容 |
| 拒答率 | 安全策略是否过严或过松 |
| 用户反馈 | 点赞、点踩、投诉、人工纠错 |
| 幻觉样本 | 需要回流修复的问题 |
| 成本 | 每个模型、租户、业务线的花费 |

更重要的是 tracing。

一次 AI 请求可能经过：

```text
用户输入 → 安全过滤 → 意图识别 → 检索 → 工具调用 → Prompt 构造 → 模型推理 → 后处理 → 返回用户
```

如果回答错了，需要知道错在哪一环。没有 tracing，排查会非常痛苦。

## 九、AI Infra 与 MLOps、LLMOps 的关系

这几个词经常混用，可以这样理解：

| 概念 | 重点 |
| --- | --- |
| MLOps | 传统机器学习模型的开发、训练、部署、监控流程 |
| LLMOps | 面向大语言模型的 Prompt、RAG、微调、评测、推理和反馈 |
| AI Infra | 更宽泛的 AI 工程底座，包含 MLOps、LLMOps、多模态、Agent、训练与推理平台 |

MLOps 更像“让机器学习模型工程化”。LLMOps 更关注大模型应用的特殊问题。AI infra 则是更底层、更全面的说法。

一个完整 AI infra 团队，可能同时负责：

1. GPU 集群和调度。
2. 数据平台和标注平台。
3. 训练框架和实验平台。
4. 模型仓库和评测平台。
5. 推理服务和模型路由。
6. RAG 平台和向量检索。
7. 线上监控、日志、反馈闭环。

## 十、AI Infra 的核心指标

AI infra 做得好不好，不能只看“有没有平台”。更重要的是指标。

| 维度 | 关键指标 |
| --- | --- |
| 训练效率 | GPU 利用率、tokens/s、训练失败率、恢复时间 |
| 实验效率 | 从想法到实验结果的时间、可复现率 |
| 数据质量 | 去重率、标注一致性、数据覆盖度 |
| 模型质量 | Benchmark、人工评测、领域任务得分 |
| 推理性能 | TTFT、TPOT、P95/P99、QPS |
| 成本 | GPU 小时、单请求成本、单 token 成本 |
| 稳定性 | 错误率、超时率、可用性 |
| 安全性 | 越狱率、危险输出率、隐私泄露率 |
| 迭代能力 | 反馈回流速度、灰度发布周期、回滚时间 |

这些指标共同决定 AI 系统能不能持续进化。

## 十一、一个 AI Infra 入门学习路线

如果你想系统学习 AI infra，可以按这个顺序：

| 阶段 | 学什么 |
| --- | --- |
| 1. 基础工程 | Linux、Docker、Kubernetes、网络、存储 |
| 2. 深度学习框架 | PyTorch、分布式训练、混合精度 |
| 3. 数据工程 | ETL、对象存储、数据版本、标注流程 |
| 4. 训练系统 | DDP、FSDP、DeepSpeed、Checkpoint |
| 5. 推理系统 | vLLM、Triton、TensorRT-LLM、KV Cache |
| 6. RAG 系统 | Embedding、向量数据库、重排、引用 |
| 7. 评测系统 | Benchmark、红队、A/B、人工评测 |
| 8. 观测系统 | Metrics、Logs、Tracing、成本监控 |
| 9. 产品闭环 | 用户反馈、数据回流、模型迭代 |

学习 AI infra 最好的方式，不是只看概念，而是亲手搭一个小闭环：

```text
准备一批领域文档
→ 做 RAG 检索
→ 接一个开源模型推理服务
→ 建一个评测集
→ 记录线上日志
→ 根据错误样本改进 prompt 或数据
→ 重新评测和发布
```

这个闭环跑通以后，再去理解大规模训练和高并发推理，会扎实很多。

## 十二、总结：AI Infra 是 AI 产品的长期竞争力

模型能力很重要，但模型不是产品的全部。

真正可用的 AI 系统，需要：

1. 稳定的数据供应。
2. 可复现的训练流程。
3. 严格的模型评测。
4. 低延迟、低成本的推理服务。
5. 完整的日志和监控。
6. 安全、合规、权限控制。
7. 持续反馈和迭代闭环。

所以，AI infra 的本质不是“基础设施”四个字听起来很底层，而是让 AI 能从一次性 demo 变成长期可运营系统的能力。

如果说模型是发动机，AI infra 就是燃油、传动、仪表盘、维修体系和道路网络。没有它，模型再强，也很难稳定地服务真实世界。
