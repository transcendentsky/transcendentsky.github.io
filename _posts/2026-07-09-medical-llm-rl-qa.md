> 一句话总结：医疗大模型用强化学习提升问答效果，核心不是让模型“更会说”，而是让它在医学正确性、证据引用、风险边界、不确定性表达和患者沟通之间学会更稳妥的取舍。

![医疗大模型的 RL 对齐流水线](/assets/images/medical-rl-qa/medical-rl-pipeline.svg)

<!--more-->

医疗问答是大模型最有价值、也最容易出风险的应用之一。

一方面，医学知识高度结构化，有指南、教材、论文、药品说明书、临床路径和大量真实病例；另一方面，医疗对话又充满不确定性：患者描述不完整，症状可能相似，疾病有轻重缓急，用药存在禁忌，很多问题不能靠一句“可能是某某病”就草率结束。

所以，医疗大模型要做好问答，不能只追求“像医生一样回答”。它还必须知道：

1. 什么时候应该回答，什么时候应该建议就医。
2. 什么时候可以解释常识，什么时候不能远程诊断。
3. 什么时候要补充缺失信息，什么时候要立即提示急症风险。
4. 什么时候要引用指南，什么时候要承认不确定。
5. 什么时候要拒绝提供危险建议，例如自行停药、加药、替代医生诊断。

这正是强化学习能发挥作用的地方。

## 一、为什么 SFT 之后还需要强化学习？

医疗大模型通常会经历几个阶段：

| 阶段 | 训练目标 | 能解决什么 | 仍然缺什么 |
| --- | --- | --- | --- |
| 预训练 | 学习通用语言和世界知识 | 具备语言理解、基础医学概念 | 未必会按医疗场景回答 |
| 医学继续预训练 | 注入医学语料、指南、论文 | 增强医学知识覆盖 | 不等于能安全对话 |
| SFT 指令微调 | 学习问答格式和任务响应 | 回答更像医疗助手 | 容易学到表面模板 |
| RLHF / DPO / Verifier-RL | 按偏好和奖励优化行为 | 更符合医生偏好和安全边界 | 奖励设计难，需防 reward hacking |

SFT 的本质是“模仿答案”。如果训练集中医生答案写得好，模型就学会类似风格；如果数据覆盖不够，模型也会照样模仿缺陷。

但医疗问答不是只看答案像不像。比如下面两个回答：

| 用户问题 | 回答 A | 回答 B |
| --- | --- | --- |
| “胸痛伴出汗，是不是胃病？” | “可能是胃食管反流，可以先吃点抑酸药观察。” | “胸痛伴出汗需要警惕心梗等急症，建议立即就医或拨打急救电话；在医生评估前不要自行判断为胃病。” |

回答 A 语言自然，也给了一个常见解释，但在医疗安全上很危险。回答 B 可能没直接满足用户“是不是胃病”的期待，却更符合真实医疗风险控制。

强化学习的价值就在这里：它可以把“医生更偏好的回答”转成训练信号，让模型不仅学会医学内容，还学会医学问答里的优先级。

## 二、医疗问答里，什么才是“好答案”？

![医疗问答奖励函数设计](/assets/images/medical-rl-qa/reward-design.svg)

普通聊天模型的偏好维度通常是有帮助、无害、诚实、表达自然。医疗问答需要更细。

| 维度 | 好答案应该做到 | 坏答案常见问题 |
| --- | --- | --- |
| 医学正确性 | 符合指南、病理机制和临床常识 | 事实错误、错误用药、错误解释检查结果 |
| 证据可追溯 | 能说明依据，必要时引用指南或检查建议 | 编造文献、给出无来源的绝对结论 |
| 风险识别 | 识别急症、禁忌、高危人群 | 延误就医，把高危症状当普通问题 |
| 不确定性表达 | 说明可能性、缺失信息和下一步检查 | 过度自信，远程确诊 |
| 个体化程度 | 根据年龄、孕产、基础病、用药史调整建议 | 一套模板回答所有人 |
| 沟通质量 | 清晰、同理、分层、可执行 | 吓唬患者、含糊其辞、没有下一步行动 |
| 安全边界 | 拒绝危险请求，建议医生复核 | 鼓励自行停药、加药、替代医生 |

因此，医疗 RL 的奖励函数不能只有一个“回答是否看起来专业”的总分。更合理的做法，是把奖励拆成多个维度，让医生、药师、医学专家或经过训练的标注员分别评价。

一个简化的奖励可以写成：

```text
总奖励 =
  医学正确性
+ 证据质量
+ 风险识别
+ 不确定性校准
+ 沟通质量
- 幻觉惩罚
- 危险建议惩罚
- 过度诊断惩罚
```

这里最重要的是负奖励。医疗场景里，很多错误不是“答得不够好”，而是“答得太肯定、太危险”。

## 三、RLHF：让医生偏好进入模型行为

RLHF 的经典流程来自 InstructGPT 一类工作：先用人工示范做 SFT，再收集人类对多个候选回答的偏好排序，训练 reward model，最后用 PPO 等强化学习方法优化模型。

放到医疗问答里，流程大致是：

| 步骤 | 医疗场景里的做法 |
| --- | --- |
| 1. 构造问题集 | 症状咨询、检查解读、用药咨询、慢病管理、分诊建议、健康科普 |
| 2. 生成多个候选回答 | 由当前模型采样多个不同回答 |
| 3. 医生偏好排序 | 医生比较回答的准确性、安全性、完整性、沟通质量 |
| 4. 训练奖励模型 | 让 reward model 预测医生更偏好的回答 |
| 5. 强化学习优化 | 用 PPO/GRPO 等方法让模型更倾向生成高奖励回答 |
| 6. 医学评测与安全评测 | 用 MedQA、PubMedQA、真实长问答、红队问题等测试 |

相比 SFT，RLHF 的优势是它不要求标注者写出完美答案，只要能比较两个答案哪个更好。这在医疗场景很有价值，因为写一个完整标准答案很费时，但医生判断“哪个回答更安全、更符合临床”通常更容易。

不过 RLHF 也有明显挑战：

1. 医生偏好成本高，且不同医生之间可能不一致。
2. 奖励模型可能学到表面特征，比如“越长越专业”。
3. PPO 训练不稳定，超参数敏感。
4. 模型可能 reward hacking，学会迎合奖励而不真正变可靠。

所以医疗大模型不能把 RLHF 当魔法。它只是把专业偏好注入模型的一种方式，前提是偏好数据和奖励模型本身足够可靠。

## 四、DPO：更轻量的偏好优化路线

DPO 的思路是：既然我们已经有“优选回答”和“劣选回答”的偏好对，就不一定非要先训练一个显式 reward model，再跑复杂 RL。DPO 可以把偏好学习转成一个更稳定的分类式优化问题。

在医疗场景里，DPO 很适合下面几类数据：

| 偏好对类型 | 优选回答 | 劣选回答 |
| --- | --- | --- |
| 急症识别 | 提示胸痛、意识障碍、呼吸困难等需要紧急就医 | 当成普通健康咨询 |
| 用药安全 | 提醒禁忌、相互作用、遵医嘱 | 直接建议剂量调整 |
| 证据引用 | 说明依据和不确定性 | 编造来源或绝对化 |
| 检查解读 | 说明指标意义和下一步检查 | 直接远程确诊 |
| 患者沟通 | 清晰、温和、可执行 | 恐吓式或含糊回答 |

DPO 的优点是实现简单、训练稳定、成本更低。对于很多医疗问答助手，DPO 可能比完整 PPO-RLHF 更适合作为第一阶段偏好对齐方案。

但 DPO 也不是万能的。它依赖偏好对质量，如果偏好数据只覆盖常见问题，模型在罕见病、复杂共病、多药联用、高风险症状上仍然可能表现不稳。

## 五、Verifier-RL：让可验证医学问题推动复杂推理

近两年一个重要变化是：强化学习不只用于“回答风格对齐”，还开始用于“复杂推理能力提升”。

HuatuoGPT-o1 这类工作就体现了这一点。它的核心思路是把一部分医学问题设计成可验证任务：模型不仅要给出答案，还要经过医学 verifier 检查输出是否正确。然后用 verifier 作为奖励信号，进一步强化模型的复杂医学推理能力。

这和传统 RLHF 有一点不同：

| 路线 | 奖励来源 | 更适合解决 |
| --- | --- | --- |
| RLHF | 医生或人类偏好 | 回答是否安全、完整、符合医生偏好 |
| DPO | 优劣回答偏好对 | 轻量偏好对齐 |
| Verifier-RL | 可验证答案、医学检查器 | 复杂推理、选择题、结构化医学问题 |
| RAG + 反馈 | 检索证据是否支持答案 | 减少幻觉、增强证据链 |

医学推理和数学推理不同。数学题答案可验证，医学问题常常没有唯一答案，而且需要考虑患者背景、概率和风险。因此医疗 verifier 的设计很难。

但它仍然很有前景。比如以下问题更适合 verifier：

1. 执业医师考试类单选题。
2. 病例摘要中的诊断倾向排序。
3. 药物相互作用判断。
4. 指标异常分级。
5. 指南路径中的下一步检查选择。

如果把这些任务设计好，RL 就能推动模型学会更可靠的医学推理路径，而不是只学会“医学口吻”。

## 六、医疗 RL 训练数据怎么构建？

医疗大模型最难的不是算法，而是数据。

一个高质量医疗 RL 数据集，至少应该覆盖这些层次：

| 数据层 | 示例 | 用途 |
| --- | --- | --- |
| 医学知识问答 | 疾病机制、检查意义、治疗原则 | 基础知识能力 |
| 患者咨询 | 症状描述、用药疑问、检查报告 | 真实对话能力 |
| 高风险红队 | 胸痛、卒中、过敏、儿童用药、孕产 | 安全边界 |
| 偏好排序 | 多个回答由医生排序 | 训练 reward model / DPO |
| 结构化 verifier | 标准答案、指南路径、药物禁忌 | Verifier-RL |
| 反例数据 | 幻觉文献、错误剂量、过度诊断 | 负奖励和安全训练 |
| 长程对话 | 多轮追问、补充病史、风险变化 | 会话级一致性 |

医疗偏好标注最好不要只让医生打一个总分，而是分项评价：

| 评分项 | 评价问题 |
| --- | --- |
| 正确性 | 是否符合医学事实和指南？ |
| 完整性 | 是否覆盖关键风险和必要下一步？ |
| 安全性 | 是否避免危险建议？ |
| 证据性 | 是否说明依据，是否避免编造？ |
| 可理解性 | 患者能否理解并执行？ |
| 校准度 | 是否适当表达不确定性？ |

这样训练出的奖励模型更不容易被“流畅、冗长、看起来专业”的回答欺骗。

## 七、强化学习具体优化了什么能力？

如果奖励设计得好，强化学习能在医疗问答中改善几个关键能力。

### 1. 从“直接回答”变成“先分诊”

用户问：

```text
我胸口闷、出汗、左肩也疼，是不是焦虑？
```

普通模型可能会解释焦虑症状。强化学习后的医疗模型应该先识别急症风险，提示心梗等可能性，建议立即就医。

这不是知识问题，而是行为优先级问题。RL 比 SFT 更适合优化这种偏好。

### 2. 从“给结论”变成“给条件化建议”

医疗答案经常不能只给一个结论，而要分条件：

| 情况 | 回答策略 |
| --- | --- |
| 症状轻微、无高危因素 | 给出观察、生活方式和就诊建议 |
| 有红旗症状 | 立即就医 |
| 已有诊断和处方 | 不擅自改药，建议咨询医生 |
| 信息不足 | 询问年龄、病史、用药、持续时间等 |

RL 可以奖励这种分层回答，而不是奖励单一结论。

### 3. 从“会背知识”变成“会承认不知道”

医疗模型最需要学会的一句话是：**仅凭这些信息无法确诊。**

好的医疗问答不是永远给答案，而是在证据不足时明确说明：

1. 目前只能提供一般信息。
2. 需要结合体征、检查和医生面诊。
3. 如果出现某些症状应立即就医。

这类“不确定性校准”很适合通过偏好优化强化。

### 4. 从“流畅”变成“可追溯”

RL 可以奖励模型引用可靠来源、说明依据、区分指南和推断。

例如，回答用药问题时，模型应该更倾向：

1. 说明药物类别和常见用途。
2. 提醒禁忌和相互作用。
3. 建议遵循处方，不自行调整剂量。
4. 对特殊人群提示医生复核。

而不是直接给一个看似确定的治疗方案。

## 八、医疗问答上线时，RL 还不够

![医疗问答上线后的安全闭环](/assets/images/medical-rl-qa/safety-loop.svg)

医疗大模型不能只靠训练阶段保证安全。上线后还需要产品和工程闭环。

| 环节 | 关键设计 |
| --- | --- |
| 输入分诊 | 识别急症、儿童、孕产、老人、精神危机等高风险场景 |
| 检索增强 | 优先接入指南、药品说明书、医院知识库 |
| 输出约束 | 禁止远程确诊、禁止自行改药、明确非替代医生 |
| 人工复核 | 高风险回答进入医生审核或延迟回复 |
| 日志审计 | 记录问题、答案、证据、模型版本、触发规则 |
| 反馈回流 | 把医生纠错和用户反馈变成偏好数据 |
| 持续评测 | 定期跑红队、医学 benchmark、真实场景回放 |

WHO 在 2025 年关于健康领域大规模多模态模型的指南中，也强调这类模型会进入医疗保健、科研、公卫和药物研发等场景，因此伦理治理、风险控制和责任边界必须前置。

换句话说，RL 可以改善模型行为，但医疗系统的可靠性来自“模型 + 工具 + 流程 + 人”的组合。

## 九、一个可落地的医疗 RL 技术方案

如果要真正做一个医疗问答大模型，我会建议按下面路线走：

| 阶段 | 目标 | 产物 |
| --- | --- | --- |
| 1. 医学 SFT | 让模型掌握基本问答格式 | 医学指令模型 |
| 2. RAG 接入 | 降低幻觉，增强证据引用 | 指南/药品/知识库检索 |
| 3. 偏好数据 | 让医生比较多候选回答 | 医疗偏好对 |
| 4. DPO 对齐 | 低成本提升回答偏好 | 第一版安全对齐模型 |
| 5. Reward Model | 多维度评分医学回答 | 医疗奖励模型 |
| 6. PPO/GRPO | 优化复杂行为策略 | 强化学习版模型 |
| 7. Verifier-RL | 提升可验证医学推理 | 推理增强模型 |
| 8. 红队与临床评估 | 检验安全边界 | 上线门槛报告 |
| 9. 人工复核闭环 | 收集真实纠错反馈 | 持续迭代数据 |

这里的关键不是一步到位，而是分阶段。

对于大多数团队，最现实的顺序是：

```text
医学 SFT → RAG → DPO → 安全红队 → 小范围上线 → 医生反馈 → RLHF / Verifier-RL
```

不要一开始就追求复杂 PPO。医疗场景里，稳定、可解释、可追溯通常比“模型更聪明一点”更重要。

## 十、必须警惕的几个坑

强化学习可以让医疗大模型变好，也可能让它变得更会“讨好奖励”。

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| Reward hacking | 模型学会写长答案、堆术语、看起来专业 | 奖励模型加入事实核验和简洁性约束 |
| 过度自信 | 对模糊症状给确定诊断 | 奖励不确定性表达，惩罚远程确诊 |
| 幻觉引用 | 编造指南、论文、药名 | 强制引用检索来源，做 citation check |
| 安全遗漏 | 没识别急症和禁忌 | 红旗症状专项评测和规则兜底 |
| 偏见与不公平 | 对年龄、性别、地区、语言表现不同 | 分层评测和公平性审计 |
| 医生偏好不一致 | 奖励模型学到噪声 | 多专家标注、一致性校验、仲裁机制 |
| 真实场景漂移 | 线上问题和训练集不同 | 持续监控和反馈回流 |

特别要注意：医疗问答不是考试答题。MedQA、PubMedQA 等 benchmark 很有用，但不能代表真实患者咨询。真实患者会省略信息、混合多个症状、表达焦虑、带着错误前提提问。RL 训练必须覆盖这些真实对话特征。

## 十一、结论：RL 的目标不是替代医生，而是校正模型行为

医疗大模型通过强化学习提升问答效果，本质上是在做三件事：

1. **把医生偏好显式化**：什么回答更安全、更完整、更可信。
2. **把风险边界训练进去**：什么时候应该拒答、转诊、提示急症。
3. **把复杂推理变得可验证**：用 verifier、指南和结构化评测减少胡乱推理。

但它不能绕开医疗系统本身的责任边界。

一个可靠的医疗问答大模型，不应该被设计成“AI 医生”，而应该被设计成：

| 角色 | 应该做 | 不应该做 |
| --- | --- | --- |
| 医学知识助手 | 解释疾病、检查、药物、健康管理常识 | 替代医生做诊断 |
| 分诊辅助 | 识别风险，提示何时就医 | 承诺某病一定没事 |
| 医生效率工具 | 辅助整理病史、查指南、生成初稿 | 跳过医生审核直接给高风险处置 |
| 患者沟通助手 | 用通俗语言解释专业问题 | 鼓励自行停药、加药 |

强化学习能让模型更接近这些目标。它不是让模型“更会说漂亮话”，而是让模型在医学知识、临床风险和人类偏好之间，学会更负责任地回答。
