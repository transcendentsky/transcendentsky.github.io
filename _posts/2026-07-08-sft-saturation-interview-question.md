# 面试题：如何判断大模型的 SFT 已经做到头了？

这道题考察的不是“会不会跑 SFT”，而是能不能判断 SFT 的边际收益、问题归因和下一阶段该不该切到 RL、数据工程、RAG 或产品约束。

一句话答案：

> SFT 做到头，不是训练 loss 降不动，而是在高质量验证集和真实业务流量上，继续增加同类 SFT 数据、训练步数或调参，已经不能稳定提升目标行为，剩余问题主要来自偏好权衡、探索决策、多步策略、奖励对齐或环境反馈。

## 一、SFT 做到头的核心指标是什么？

**答案：**

核心指标不是单一的 training loss，而是“目标行为的边际收益是否接近耗尽”。可以从四类指标判断。

### 1. 业务主指标进入平台期

例如：

- 任务成功率、准确率、召回率、pass@k、人工验收通过率不再提升。
- 线上 badcase 率下降变慢。
- A/B 测试中用户满意度、解决率、转人工率没有显著改善。
- 继续加数据或训练轮数，只带来 0.x% 的波动，且置信区间内不显著。

面试中可以强调：SFT 是否做到头，要看独立验证集、人工评测集和线上指标，而不是只看训练集。

### 2. 泛化指标不再提升，甚至开始变差

如果训练集继续变好，但验证集、跨领域集、难例集不变或下降，说明模型更多是在记忆样本和模板。

典型表现：

- 训练 loss 下降，验证 loss 持平或上升。
- 标准 benchmark 小幅下降。
- 模型回答越来越像训练集模板，面对新表达泛化差。
- 对长尾输入、复杂上下文、多轮场景的鲁棒性没有提升。

### 3. 人工偏好评测达到 SFT 能解决的上限

SFT 擅长学习“给定输入到标准输出”的映射，但不擅长在多个合理答案之间优化偏好排序。

如果人工评测发现：

- 答案都基本正确，但风格、详略、风险取舍不稳定。
- chosen 和 rejected 都能由 SFT 数据解释，差异更多是偏好问题。
- 用户对“更有帮助”“更谨慎”“更简洁”的偏好很难写成唯一标准答案。

这时继续 SFT 的收益通常有限，应该考虑 DPO、PPO、GRPO 等偏好优化或 RL 方法。

### 4. 问题来源不再是“不会模仿”，而是“不会选择”

SFT 的目标是最大化参考答案似然，本质上让模型模仿示范数据。它适合解决：

- 格式不会。
- 指令不跟。
- 领域表达不熟。
- 工具调用样式不稳定。
- 基础任务流程没学会。

如果剩余问题变成：

- 多个动作都看似合理，但需要长期收益判断。
- 需要探索、试错、环境反馈。
- 需要在安全性、帮助性、简洁性之间做动态取舍。
- 需要根据最终结果而不是单步示范来优化。

这就是 SFT 接近边界的信号。

## 二、SFT 还没做到头的信号有哪些？

**答案：**

SFT 还没做到头，通常意味着问题仍然可以通过更好的监督数据、更合理的训练配方或更细的任务拆解解决。

### 1. 明显的指令遵循和格式错误

例如：

- 让输出 JSON，却经常多输出解释文本。
- 要求只返回结论，却重复背景说明。
- 系统提示优先级不稳定。
- 多轮对话中忘记用户约束。

这些通常仍是 SFT 的强项。优先补高质量格式样本、反例修正样本和多轮样本。

### 2. 错误集中在可标注、可示范的模式上

如果 badcase 可以被稳定归类，例如：

- 某类实体抽取漏字段。
- 某类工具参数填错。
- 某类行业术语解释不规范。
- 某类 SQL、代码、报告模板生成错误。

这说明监督信号还不充分，SFT 仍有空间。

### 3. 数据覆盖明显不足

典型信号：

- 训练集中缺少长尾领域、边界条件、异常输入。
- 问法多样性不足，模型只会处理固定模板。
- 多轮任务中缺少澄清、拒答、追问、工具失败等场景。
- 高质量样本太少，低质量样本太多。

这种情况下，先做数据治理比切 RL 更有效。

### 4. 验证集仍随数据质量提升而明显改善

如果你清洗数据、补难例、增加覆盖后，验证集和人工评测还在稳定提升，说明 SFT 还没有到头。

一个实用判断：

```text
同类高质量数据翻倍后，核心指标还能显著上涨 -> SFT 还没到头
同类高质量数据翻倍后，核心指标基本不动 -> 接近 SFT 上限
```

### 5. 训练配方还没有被认真调过

有些团队误以为 SFT 到头，其实是训练没有做好。

需要排查：

- 学习率是否过大或过小。
- epoch 是否过多导致过拟合。
- loss mask 是否正确，只训练 assistant 输出。
- 长上下文是否被截断。
- chat template 是否和推理一致。
- LoRA rank、target modules、batch size 是否合理。
- 是否混入了错误标签、重复样本、污染样本。

如果这些基础项没排清，不应轻易判断 SFT 已经到头。

## 三、SFT 已经做到头的信号有哪些？

**答案：**

SFT 做到头通常会同时出现“指标平台期、继续训练副作用增加、问题类型转移”三个特征。

### 1. 继续训练只改善训练集，不改善真实任务

表现为：

- train loss 继续下降。
- validation loss 不变或变差。
- 人工评测胜率不再提升。
- 线上真实请求没有改善。
- 新增数据只让模型更像标注答案，但不更有用。

这说明模型已经学到当前 SFT 数据能提供的大部分监督信号。

### 2. 加数据的边际收益快速下降

例如前 1 万条高质量样本提升很明显，后面再加 5 万条相似样本几乎没有提升。

如果新增数据只是同分布、同模板、同难度，模型会很快吃完收益。此时继续堆量可能不如：

- 重新设计评测集。
- 针对难例做主动采样。
- 提升标注一致性。
- 引入偏好数据。
- 切到 RL 或 DPO。

### 3. 模型变得更模板化，创造性和鲁棒性下降

过度 SFT 常见副作用：

- 回答套话增多。
- 遇到不确定问题也强行给标准答案。
- 拒答策略僵硬。
- 对 prompt 变化敏感。
- 原模型通用能力下降。

这说明 SFT 继续推进可能在牺牲基础能力。

### 4. 剩余 badcase 难以写出唯一标准答案

如果 badcase 的修复不再是“给一个正确答案”，而是要判断：

- 哪个答案用户更喜欢。
- 哪种表达更符合风险偏好。
- 是否应该追问、拒答、调用工具或直接回答。
- 当前动作对后续多轮任务是否更好。

这类问题往往不适合只靠 SFT。

### 5. 单步模仿已经足够，但多步结果仍不好

例如 Agent、代码修复、工具调用、数学推理、检索问答中，单步输出看起来都合理，但最终任务失败。

原因是 SFT 优化的是局部示范，不直接优化最终结果。此时需要：

- 过程监督。
- 结果奖励。
- 执行环境反馈。
- 搜索或自我修正。
- RL 或基于偏好的优化。

## 四、SFT 应该怎么做？

**答案：**

SFT 的正确做法不是盲目堆数据，而是围绕目标行为建立“数据、训练、评估、迭代”的闭环。

### 1. 先定义目标行为和不可接受行为

训练前要明确：

- 模型要解决什么任务。
- 输出格式是什么。
- 哪些场景必须拒答。
- 哪些场景必须追问。
- 成功和失败如何评估。
- 是否要保留原模型通用能力。

没有评估标准的 SFT 很容易变成“训练完感觉更像了”。

### 2. 建高质量数据，而不是只追求数量

优先数据类型：

- 真实业务请求。
- 高价值高频场景。
- 历史 badcase 修复样本。
- 边界条件和异常输入。
- 多轮对话样本。
- 工具调用成功和失败样本。
- 拒答、澄清、降级处理样本。

数据质量要求：

- 标注一致。
- 事实正确。
- 结构清晰。
- 覆盖足够多样。
- 避免重复样本。
- 避免把 prompt hack、错误答案和脏格式训练进去。

### 3. 训练时控制过拟合和灾难性遗忘

常见策略：

- 使用较小学习率。
- 控制 epoch，不追求训练 loss 最低。
- 做 early stopping。
- 混入少量通用指令数据。
- 优先 LoRA/QLoRA 做低成本迭代。
- 保持训练 chat template 和推理模板一致。
- 对 assistant response 做 loss，避免错误训练 user/system 内容。

### 4. 评估集要分层

至少准备四类评估：

- In-domain 集：目标业务主场景。
- Hard case 集：历史失败和长尾难例。
- General capability 集：防止通用能力掉太多。
- Safety/format 集：检查拒答、合规和结构化输出。

评估方式可以组合：

- 自动指标。
- 规则校验。
- LLM-as-judge。
- 人工双盲偏好评测。
- 小流量线上 A/B。

### 5. 迭代优先级

一个推荐顺序：

```text
定义评估目标
-> 清洗和构造 SFT 数据
-> 小规模 LoRA 试验
-> 分层评估和 badcase 归因
-> 补难例与修数据
-> 判断边际收益
-> 再决定是否切 DPO/RL/RAG/规则约束
```

面试时可以补一句：SFT 的工程核心是 badcase driven，不是 train loss driven。

## 五、SFT 和 RL 的边界是什么？

**答案：**

最简洁的区分：

> SFT 适合让模型学会“怎么做”；RL 适合让模型学会“做哪个更好”。

### 1. SFT 适合的场景

SFT 更适合有明确示范答案的问题：

- 指令遵循。
- 输出格式。
- 领域术语和表达。
- 固定任务流程。
- 工具调用格式。
- 标准问答、抽取、分类、改写、摘要。

如果可以写出稳定、唯一或近似唯一的高质量参考答案，优先 SFT。

### 2. RL 适合的场景

RL 更适合只有结果好坏、偏好高低或长期收益的问题：

- 多轮 Agent 任务。
- 工具使用策略。
- 代码执行反馈。
- 数学推理最终正确性。
- 安全性和帮助性的权衡。
- 用户偏好优化。
- 长链路任务成功率优化。

如果很难写出唯一标准答案，但可以判断哪个结果更好，考虑 DPO/RLHF/RLAIF/GRPO 等方法。

### 3. 二者不是替代关系，而是阶段关系

常见训练路径：

```text
预训练模型
-> SFT：学会指令、格式、任务基本能力
-> Preference Optimization：学习人类偏好
-> RL：用奖励或环境反馈优化最终目标
```

没有 SFT 直接做 RL，通常探索成本高、输出不稳定。SFT 做得太过，又可能让模型分布过窄，限制 RL 探索。

### 4. 判断是否从 SFT 切到 RL 的实用标准

可以用下面的问题判断：

- 问题是否已经不是“模型不会输出正确格式”？
- 是否很难为每个输入写唯一标准答案？
- 是否可以比较两个答案哪个更好？
- 是否可以定义奖励或获得环境反馈？
- 是否需要优化最终任务成功率，而不是单步模仿？
- 继续 SFT 是否出现明显平台期或副作用？

如果答案多数是“是”，就该考虑从纯 SFT 转向偏好优化或 RL。

## 六、面试中的高质量回答模板

**答案模板：**

我不会只看 SFT 的训练 loss 来判断是否做到头，而会看目标任务上的边际收益。具体来说，我会准备独立验证集、难例集、通用能力回归集和线上 A/B 指标。如果继续增加同分布高质量 SFT 数据、训练步数或调学习率后，训练集还在变好，但验证集、人工偏好胜率、线上成功率都不再显著提升，甚至出现模板化、泛化下降和通用能力退化，就说明 SFT 接近上限。

如果 badcase 仍然集中在格式错误、指令不跟随、工具参数错误、领域表达不规范这类可标注、可示范的问题上，我会继续做 SFT，重点补高质量难例和清洗数据。如果剩余问题变成多个答案都对但偏好不同、多步策略选择、工具调用时机、长期任务成功率、安全和帮助性的权衡，那就不是 SFT 最擅长的问题了，应该考虑 DPO、RLHF、GRPO 或基于环境反馈的 RL。

所以 SFT 和 RL 的边界在于：SFT 优化模仿示范，解决“会不会做”；RL 优化奖励和偏好，解决“怎么选更好、最终结果更好”。SFT 是对齐和任务适配的基础，但不能把所有问题都压到 SFT 上。

## 七、常见追问

### 1. SFT loss 降不动是否等于做到头？

不等于。loss 降不动可能是学习率、batch size、数据噪声、模型容量、训练实现的问题。SFT 是否做到头要看目标评估指标和边际收益。

### 2. 验证 loss 上升但人工评测变好，怎么判断？

要看验证集是否代表真实目标。如果人工评测和线上指标更可靠，说明 loss 不一定是好代理指标。但也要警惕过拟合评测偏好，最好用多套评估交叉验证。

### 3. 为什么不能一直用 SFT 替代 RL？

因为 SFT 只能学习数据中给出的示范，不直接优化偏好、长期收益和环境结果。对于没有唯一答案、需要策略选择的问题，SFT 会把复杂偏好压成单一参考答案，信号效率低且容易学歪。

### 4. SFT 已经到头后一定要做 RL 吗？

不一定。还可能需要：

- RAG：解决知识缺失和事实更新。
- 规则或约束解码：解决严格格式。
- 工具系统改造：解决外部能力不足。
- 数据重构：解决标注标准错误。
- 产品交互调整：解决需求本身不清晰。

RL 只适合奖励可定义、反馈可获得、成本可接受的场景。

---

## 八、总结

判断 SFT 是否做到头，关键看三件事：

- **边际收益**：继续加同类数据和训练是否还能显著提升目标指标。
- **问题类型**：剩余问题是可示范的监督学习问题，还是偏好、策略和长期收益问题。
- **副作用**：是否开始出现过拟合、模板化、泛化下降和通用能力退化。

面试中最好的回答不是“loss 不降了”，而是：

> SFT 做到头的标志，是监督示范信号的边际价值耗尽；继续 SFT 不能稳定改善真实目标，剩余收益需要来自更好的反馈信号、偏好优化、RL、RAG 或系统工程。
