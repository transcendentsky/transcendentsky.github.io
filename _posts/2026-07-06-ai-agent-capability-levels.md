> 一句话总结：AI Agent 的能力分级，不是给产品贴“高级”标签，而是判断一个系统在多大范围内能够理解目标、拆解任务、调用工具、处理异常，并在人的监督下稳定交付结果。

![AI Agent 能力分级 L0-L5](/assets/images/agent-levels/agent-capability-ladder.svg)

<!--more-->

过去一年，“Agent”几乎成了 AI 产品的标准词汇。聊天机器人叫 Agent，工作流自动化叫 Agent，能调用几个 API 也叫 Agent，甚至一些固定脚本换个自然语言入口，也会被包装成智能体。

这会带来一个问题：如果所有东西都叫 Agent，我们就很难判断它到底能做什么、风险在哪里、应该由谁负责。

所以，比起问“这是不是 Agent”，更有价值的问题是：它处在什么能力等级？它能不能自己理解目标？能不能规划步骤？能不能调用工具？遇到异常时会不会停下来？它的行动边界在哪里？人类应该在什么节点介入？

本文尝试用 L0-L5 六个等级，给 AI Agent 建立一个更清晰的能力坐标系。

## 一、为什么需要 Agent 能力分级？

Agent 的核心不是“会聊天”，而是围绕目标持续行动。

一个真正意义上的智能体，通常至少包含五类能力：理解目标、感知环境、规划任务、调用工具、根据反馈调整行动。能力越强，它越能从“回答问题”走向“完成任务”。

但能力增强也意味着风险增强。一个只会生成文案的助手，出错成本通常是修改一版内容；一个能自动发邮件、改数据库、调度资金、执行代码的智能体，出错就可能直接影响真实业务。

因此，Agent 分级有三个现实意义：

1. 帮助产品和技术团队说清楚系统能力，避免把“能对话”误认为“能自治”。
2. 帮助业务方选择合适的自动化程度，不为低风险任务过度设计，也不让高风险任务失控运行。
3. 帮助团队设计治理机制，把权限、审核、日志、回滚和责任边界放在正确位置。

分级的目的不是追求最高级，而是让能力、场景、风险和监督方式匹配。

## 二、L0-L5：六个能力等级

不同研究和产业文章对 Agent 分级的命名并不完全一致，但核心维度大体相同：自治程度、任务复杂度、工具使用、记忆学习、多智能体协作和人类控制方式。参考自动驾驶 L0-L5 的表达方式，我们可以把 AI Agent 拆成以下六级。

| 层级 | 名称 | 核心能力 | 典型表现 | 主要短板 |
| --- | --- | --- | --- | --- |
| L0 | 工具自动化 | 执行固定规则 | 定时任务、RPA、固定审批流 | 不理解开放目标 |
| L1 | 对话助手 | 理解自然语言并生成建议 | 问答、总结、改写、代码片段 | 不独立行动 |
| L2 | 工具调用型 Agent | 在授权下调用外部工具 | 检索、查库、运行代码、调用 API | 工具链短，异常处理弱 |
| L3 | 任务执行型 Agent | 拆解并推进多步骤任务 | 调研、代码修改、数据分析、报告生成 | 需要强可观测和人工节点 |
| L4 | 领域自治型 Agent | 在边界内持续运行并闭环 | 运维、客服、指标监控、标准流程处理 | 对治理、回滚、审计要求高 |
| L5 | 自演化智能体 | 开放目标、多 Agent 协作、自我优化 | 高度受控的自治系统探索 | 责任、伦理、安全边界复杂 |

### L0：工具自动化，没有智能体自治

L0 本质上不是智能体，而是传统工具、脚本或固定工作流。

典型例子包括定时任务、Excel 宏、RPA 脚本、固定规则的审批流、按照模板生成报表的程序。它们可以提高效率，但不会理解开放目标，也不会自己判断下一步。

L0 的关键特征是：

- 人定义完整流程。
- 系统只执行固定规则。
- 异常通常需要人发现和处理。
- 结果质量主要取决于流程设计者。

L0 的价值仍然很大。大量企业自动化其实都停留在这个等级，而且只要规则稳定、流程清晰，L0 往往比复杂智能体更可靠。

### L1：对话助手，能理解语言但不独立行动

L1 是大多数聊天助手和内容生成工具所在的位置。

它能够理解自然语言指令，完成问答、总结、翻译、改写、头脑风暴、代码片段生成等任务。它比 L0 更灵活，因为用户不需要把流程写成规则，只要用自然语言描述需求。

但 L1 仍然主要停留在“生成建议”层面。它可以告诉你应该怎么做，却不会真正替你执行完整流程；它可以写一封邮件草稿，但不会自动选择收件人、发送邮件并追踪回复。

L1 的核心边界是：它辅助人的思考和表达，但不拥有真实行动权。

### L2：工具调用型 Agent，能完成局部动作

L2 开始具备智能体雏形。

它不仅能回答问题，还能在人的授权下调用工具：检索资料、查询数据库、运行代码、读取文件、调用 API、生成图表、提交表单。用户给出目标后，它可以选择一个或多个工具完成局部任务。

比如：

- 让智能体检索某个主题的最新论文并整理摘要。
- 让智能体读取一组日志文件并找出异常。
- 让智能体调用日历 API 安排一个会议草案。
- 让智能体运行测试并解释失败原因。

L2 的能力跃迁在于“从说到做”。但它通常还不能长时间自主规划复杂任务，尤其不适合在缺少人工确认的情况下连续执行高风险动作。

这一阶段最重要的工程问题是权限控制：它能用哪些工具？能读哪些数据？能写哪些系统？哪些动作必须先让人确认？

### L3：任务执行型 Agent，能规划并推进多步骤任务

L3 是许多团队真正想要的“办事型 Agent”。

用户不再只是要求它完成一个局部动作，而是给它一个较完整的任务目标。它需要自己拆解步骤、制定计划、按阶段调用工具、根据中间结果调整路径，并把阶段成果反馈给人。

例如：

- 给定一个产品需求，生成技术方案、拆分任务、修改代码、运行测试、提交变更摘要。
- 给定一个研究问题，检索资料、筛选来源、形成结构化综述、列出证据和不确定性。
- 给定一个业务指标异常，拉取数据、定位可能原因、生成分析报告和后续排查建议。

L3 的关键不只是工具调用，而是任务编排。它要知道先做什么、后做什么，什么时候继续，什么时候停下来问人。

这一等级对可观测性要求很高。人不能只看到最终答案，还要看到计划、执行记录、工具调用结果、失败重试和风险提示。否则，L3 很容易变成一个“看起来很能干、实际不可控”的黑盒。

### L4：领域自治型 Agent，能在边界内持续运行

L4 的智能体已经可以在明确边界内长期运行。

它不只是完成一次任务，而是面向一个领域持续感知状态、发现问题、规划行动、调用多个工具或多个子智能体协同完成工作。人类不再逐步指挥它，而是定义目标、规则、权限和例外处理机制。

典型场景包括：

- 运维智能体持续监控系统指标，自动定位异常、执行低风险修复、升级高风险事件。
- 客服智能体在知识库、订单系统和工单系统之间协同，独立处理标准问题，把争议问题转人工。
- 数据分析智能体定期监控业务指标，发现异常后自动生成归因分析和实验建议。

L4 的前提非常严格：领域边界清晰，数据质量稳定，工具接口可靠，风险动作可分级，系统有完整日志、回滚和审计。

也就是说，L4 不是“把人拿掉”，而是把人的角色从过程操作者升级为规则制定者、异常处理者和责任承担者。

### L5：自演化智能体，更像研究目标而不是常规产品

L5 是最容易被误解的等级。

如果说 L4 是在明确边界内自治，那么 L5 则意味着智能体能够在更开放的环境中提出子目标、重构流程、学习新策略、协调多个智能体，并持续优化自身的工作方式。

它听起来很像“数字员工”甚至“自治组织”。但在现实工程中，L5 仍然更接近研究目标或极少数高度受控场景下的探索形态，而不是可以随便交给业务系统的成熟能力。

原因很简单：开放环境里的目标往往模糊，约束会变化，价值判断难以形式化，责任边界也很难完全交给机器。越接近 L5，越不能只讨论模型能力，还必须讨论治理、伦理、安全、法律责任和组织机制。

所以对 L5 更稳妥的理解是：它代表 Agent 技术的方向，但不代表企业应该无条件追求完全自治。

## 三、分级不是看模型有多大，而是看系统能不能闭环

很多人会把 Agent 能力直接等同于大模型能力，这是一个常见误区。

模型越强，确实越容易支撑高等级 Agent，但 Agent 等级不是由模型参数量决定的，而是由整个系统闭环决定的。

判断一个 Agent 处于哪个等级，可以看五个问题：

1. 目标理解：它只是响应单条指令，还是能理解较长期的任务目标？
2. 任务规划：它是否能把目标拆成可执行步骤，并根据反馈调整计划？
3. 工具行动：它是否能调用真实工具，是否具备读写外部系统的能力？
4. 异常处理：遇到不确定性、冲突或失败时，它会盲目继续，还是会暂停并请求确认？
5. 经验沉淀：它是否能把执行记录、偏好、规则和失败经验转化为后续可复用能力？

![Agent 能力维度矩阵](/assets/images/agent-levels/capability-matrix.svg)

这五个问题比“用了哪个模型”更重要。

一个强模型套在没有权限隔离、没有日志、没有回滚的系统里，并不会自动变成高级 Agent；相反，它可能只是一个更危险的自动化入口。一个中等模型如果运行在边界清晰、工具可靠、反馈完善的系统里，反而可能稳定达到 L2 或 L3。

## 四、能力越高，人类越要换一种方式介入

Agent 分级最重要的启发之一，是人类监督方式要跟着变化。

![不同 Agent 等级下的人类控制点](/assets/images/agent-levels/human-control-loop.svg)

在 L0-L2，人主要控制步骤。你告诉系统做什么、用什么工具、执行哪一步，然后检查结果。

到了 L3-L4，人主要控制节点。智能体可以自己推进一段流程，但关键计划、关键授权、关键风险和最终交付仍然需要人确认。

到了 L5，人控制的是制度。人不可能检查每一个微动作，只能通过目标边界、策略约束、审计追责、停机机制和外部监管来控制系统。

这意味着，智能体越强，人类不是越轻松，而是监督方式更抽象、责任也更重。低等级系统需要操作手册，高等级系统需要治理架构。

一个实用原则是：

- 可逆、低风险、高频任务，可以提高自动化等级。
- 不可逆、高风险、责任敏感任务，必须保留人工确认。
- 任务边界不清、数据质量不稳、评价标准模糊时，不要急着上高自治。
- 只要 Agent 能写入真实系统，就必须有权限、日志、回滚和审计。

## 五、不同等级适合什么场景？

如果把 L0-L5 放到实际业务里，可以形成一个简单选择表。

| 等级 | 适合场景 | 不适合场景 | 人类角色 |
| --- | --- | --- | --- |
| L0 | 固定规则、批处理、稳定流程 | 开放问题、复杂判断 | 流程设计者 |
| L1 | 写作、问答、总结、创意辅助 | 自动执行、系统写入 | 内容审核者 |
| L2 | 检索、分析、代码运行、API 查询 | 连续高风险操作 | 授权者 |
| L3 | 研发辅助、研究综述、数据排查、项目型任务 | 无法观察过程的黑盒任务 | 节点审批者 |
| L4 | 运维、客服、数据监控等边界清晰领域 | 规则频繁变化且责任重的领域 | 规则制定者 |
| L5 | 研究探索、高度受控的自治系统实验 | 常规生产系统直接放权 | 治理与责任主体 |

这张表背后的核心判断是：不要用“技术能不能做”替代“组织该不该放权”。

真正成熟的 Agent 落地，不是把所有任务都推到更高等级，而是把不同任务放到合适等级。一个企业可能同时存在 L0 的报表脚本、L1 的知识助手、L2 的数据分析助手、L3 的研发智能体，以及少量 L4 的领域自治系统。

## 六、从 L2 到 L3，往往是最关键的一跳

今天大多数可用的 Agent 产品，其实集中在 L1-L3。

L1 已经非常普遍，L2 也在快速成熟：检索、代码执行、文件读写、API 调用都已经成为标准能力。真正难的是从 L2 到 L3。

因为 L3 不只是“多调几个工具”，而是要形成稳定任务闭环：

- 需求不完整时会澄清。
- 计划不可靠时会修正。
- 工具失败时会重试或换路。
- 信息冲突时会暴露不确定性。
- 关键动作前会请求授权。
- 完成后能解释自己做了什么。

这也是为什么很多 Agent 演示看起来惊艳，落到生产环境却不稳定。演示通常只展示成功路径，而真实业务充满异常路径。L3 的核心能力，恰恰是处理异常、解释过程和稳定交付。

如果一个团队要建设 Agent，最务实的路径通常不是直接追求 L5，而是先把 L2 做稳，再在少数高价值、边界清晰的任务上打磨 L3。

![从工具 Agent 到闭环自治的工程路径](/assets/images/agent-levels/agent-engineering-path.svg)

更细一点，可以把工程演进拆成下面四步：

| 阶段 | 目标 | 重点建设 | 不建议做什么 |
| --- | --- | --- | --- |
| 第一步 | L2 工具调用稳定 | 工具 schema、参数校验、权限、日志 | 不要让模型随意拼 API |
| 第二步 | L3 多步任务可控 | 任务状态、计划展示、中间产物、人工确认 | 不要黑盒长链路执行 |
| 第三步 | L4 闭环自治 | 自检、失败重试、回滚、记忆、验收标准 | 不要跳过审计和风控 |
| 第四步 | L5 组织协同 | 多 Agent 协作、资源调度、治理体系 | 不要把责任边界交给模型 |

## 七、怎么评估一个 Agent 到底处在哪一级？

分级不能只看演示效果，最好用一组可验证指标来判断。

| 指标 | 说明 | 为什么重要 |
| --- | --- | --- |
| 任务完成率 | 端到端目标是否完成 | 衡量真实业务价值 |
| 人工干预率 | 过程中需要人救场的比例 | 衡量自治程度 |
| 异常恢复率 | 工具失败、数据缺失时能否恢复 | 衡量鲁棒性 |
| 计划可解释性 | 是否能展示计划、中间状态和决策依据 | 衡量可控性 |
| 可追溯性 | 是否能还原工具调用、数据来源和结果 | 衡量治理能力 |
| 成本可控性 | token、工具、时间成本是否可控 | 衡量规模化能力 |

如果一个系统单步准确率很高，但端到端完成率低，它仍然不算高层级 Agent。如果一个系统能完成复杂任务，但过程不可观测、失败不可回滚，也不应该被放到高自治等级。

## 八、写在最后

AI Agent 的未来不会只有一种形态。

有些 Agent 会像工具，安静地完成固定任务；有些 Agent 会像助手，帮助人思考、写作、分析；有些 Agent 会像执行伙伴，能够推进复杂任务；还有少数 Agent 会在特定领域里承担持续自治职责。

L0-L5 分级的价值，不在于制造一个“谁更高级”的排行榜，而在于提醒我们：能力、权限、风险和责任必须一起设计。

真正可靠的智能体系统，不是最会说话的系统，也不是最敢自动执行的系统，而是在合适的边界内，把目标理解、任务规划、工具行动、异常处理和人类监督做成闭环的系统。

当我们能清楚说出一个 Agent 处于哪一级、为什么只能到这一级、升级到下一等级还缺什么条件时，才算真正开始理解 Agent。
